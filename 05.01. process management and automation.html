Lesson 1

Process Concepts

PID (Process Identifier), PPID (Parent Process Identifier), States — The foundation of everything.

What Is a Process?

A program is a recipe sitting in a cookbook — static instructions on paper. A process is a chef actively cooking from that recipe. One recipe can spawn multiple cooks working in parallel, just as one program (your web browser) can spawn dozens of processes simultaneously.

PID (Process Identifier)

A unique number assigned to every running process — like an employee badge. No two active processes share the same PID.

PPID (Parent Process Identifier)

The PID of the process that created this one. Every process (except PID 1) has a parent, forming a hierarchy — a process tree.

PID 1 — The Ancestor

The first user-space process (systemd or init). Always PID 1. Every other process on the system descends from it.

The Process Family Tree

Process Hierarchy Example
PID 1 (systemd) ├── PID 482 (sshd) │ └── PID 1200 (bash) │ └── PID 1350 (python server.py) ├── PID 530 (nginx master) │ ├── PID 531 (nginx worker) │ └── PID 532 (nginx worker) └── PID 610 (postgres) ├── PID 611 (postgres writer) └── PID 612 (postgres worker)

Process States — The Lifecycle

StateSymbolDescriptionReal-World Example
RunningRActively using the CPU (Central Processing Unit)Your code executing right now
Sleeping (Interruptible)SWaiting for I/O (Input/Output) — can be woken by a signalWeb server waiting for a request
Sleeping (Uninterruptible)DDeep I/O wait — cannot be interruptedDisk read in progress
StoppedTPaused by signal (SIGSTOP) or debuggerYou pressed Ctrl+Z
ZombieZFinished but parent hasn't called wait()Ghost in the process table

The Zombie Problem

Analogy

An employee who has finished a task standing at the manager's desk, report in hand, waiting to be dismissed. Until the manager (parent) accepts the report (wait()), the employee just stands there — occupying a slot on the roster but doing no actual work.

In a badly written distributed application, zombie accumulation can exhaust the system's process table — a real-world outage scenario. Fix: always ensure parent processes reap their children.

Task: This Bash script forks child processes, prints their PID and PPID, shows their states, and creates a deliberate zombie to illustrate the concept.

Bash
#!/usr/bin/env bash
# Demonstrate PID, PPID, and process states

echo "[Parent] My PID is: $$"
echo "[Parent] My PPID is: $PPID"

# Fork a well-behaved child process
(
    echo "[Child-A] My PID is: $BASHPID"
    echo "[Child-A] My PPID is: $$"
    sleep 1
    echo "[Child-A] Finished. Exiting."
) &
CHILD_A_PID=$!

# Create a zombie on purpose
(
    echo "[Child-B] Exiting immediately."
    exit 0
) &
CHILD_B_PID=$!

sleep 2

# Show process states — look for "Z" (zombie)
ps -o pid,ppid,stat,comm -p $CHILD_A_PID -p $CHILD_B_PID

# Reap all children
wait
echo "All children reaped. No more zombies."

Step-by-Step

  1. $$ gives the current script's PID; $PPID gives its parent.
  2. Parentheses ( ... ) & create a subshell (child process) running in the background.
  3. Child-B exits immediately but the parent sleeps — creating a zombie (state Z).
  4. ps -o pid,ppid,stat,comm shows the state column where you can spot the zombie.
  5. wait reaps all children, clearing the zombies from the process table.
Process State Lifecycle Diagram
CREATED ──dispatch──► RUNNING (R) │ ┌─────────────┼──────────────┐ ▼ ▼ ▼ SLEEPING (S/D) STOPPED (T) ZOMBIE (Z) wait for I/O paused by done, parent or event signal hasn't reaped │ │ │ ▼ ▼ ▼ RUNNING (R) RUNNING (R) TERMINATED (event ready) (SIGCONT) (parent wait())
Key Takeaway

Every process has a PID, a parent (PPID), and moves through defined states (R, S, D, T, Z). Understanding this hierarchy is fundamental to managing distributed services, debugging hangs, and preventing resource exhaustion from zombie processes.

Lesson 2

Signals

SIGTERM, SIGKILL, SIGHUP — The kernel's messaging system.

If processes are employees working in separate offices, signals are the internal memo system. Each signal is a number with a name, defined by POSIX (Portable Operating System Interface). About 30 standard signals exist. Three are essential:

SIGTERM — Signal 15

"Please shut down gracefully." The process CAN catch this signal. Best practice: save state, close connections, notify peers, exit. Used by Kubernetes during rolling deployments.

Analogy

Professor says: "Class ends in 5 minutes — save your notes."

SIGKILL — Signal 9

"Terminate immediately." The process CANNOT catch, handle, or ignore this. The kernel kills it directly. Last resort only — no cleanup possible.

Analogy

The fire alarm dumps water on everything. Nobody saves their work.

SIGHUP — Signal 1

"Reload your configuration." Originally meant "terminal disconnected." Modern usage: tell a daemon to re-read its config with zero downtime. Used by NGINX.

Analogy

Sliding an updated menu under the kitchen door — chefs don't stop cooking.

Signal Delivery Mechanism

How Signals Travel
SENDER KERNEL TARGET PROCESS (user, process, Marks signal Signal handler runs or kernel) as "pending" at next scheduling slot │ │ │ └──── kill() ─────►└──── pending ───────►│ │ ┌────────────┴─────────────┐ ▼ ▼ ▼ SIGTERM(15) SIGKILL(9) SIGHUP(1) Catchable Uncatchable Catchable Graceful Immediate Config shutdown termination reload

Task: A long-running service that handles SIGTERM gracefully, uses SIGHUP to reload config, and demonstrates SIGKILL cannot be caught.

Bash
#!/usr/bin/env bash
# Simulate a service handling SIGTERM, SIGHUP, and SIGKILL

# Configuration variable (simulates a real config value)
CONFIG_VALUE="initial-setting"

# Graceful shutdown handler for SIGTERM
handle_sigterm() {
    echo "SIGTERM received. Saving state..."
    echo "last_config=$CONFIG_VALUE" > /tmp/state.txt
    echo "Goodbye. Exiting cleanly."
    exit 0
}

# Config reload handler for SIGHUP
handle_sighup() {
    CONFIG_VALUE="reloaded-$(date +%s)"
    echo "SIGHUP: Config reloaded → $CONFIG_VALUE"
}

# Register signal handlers with trap
trap handle_sigterm SIGTERM
trap handle_sighup SIGHUP

echo "Service PID: $$  |  Config: $CONFIG_VALUE"
echo "Try: kill -SIGTERM $$  |  kill -SIGHUP $$  |  kill -9 $$"

# Main heartbeat loop
while true; do
    echo "Heartbeat | uptime=${SECONDS}s | config=$CONFIG_VALUE"
    sleep 3
done

Step-by-Step

  1. trap handle_sigterm SIGTERM registers a custom handler function for signal 15.
  2. When SIGTERM arrives, the function saves state to a file and exits with code 0.
  3. SIGHUP triggers a config reload — the variable changes without restarting the service.
  4. SIGKILL (kill -9) bypasses all handlers — the kernel terminates immediately.
Key Takeaway

SIGTERM (15) = graceful shutdown (catchable). SIGKILL (9) = forced kill (uncatchable). SIGHUP (1) = config reload (catchable). Always prefer SIGTERM over SIGKILL in production. Graceful shutdown prevents data corruption, lock leaks, and orphaned connections in distributed systems.

Lesson 3

Process Monitoring

ps, top, htop, pidof, pgrep — Your system's vital signs.

In a distributed system with hundreds of microservices, knowing what is running, how much CPU (Central Processing Unit) and RAM (Random Access Memory) it consumes, and whether it is healthy is non-negotiable. Think of it as a hospital's vital-signs monitors — but for your operating system.

ps — Snapshot Camera

One-time photo of running processes. Does not update.

ps aux — All users, full format
ps -o pid,ppid,stat,%cpu — Custom columns

Best for: Scripts, piping to other commands

top — Live Dashboard

Continuous real-time view. Built into every Linux distro.

Shortcuts: M=memory, P=CPU, k=kill, q=quit

Best for: Quick resource checks

htop — Modern Dashboard

Color-coded CPU/RAM bars, mouse support, tree view. Install: sudo apt install htop

Best for: Daily interactive monitoring

pidof — Quick Lookup

Fast name → PID(s). pidof nginx → 1842 1843

Best for: "Is X running?" checks

pgrep — Regex Search

Pattern-matching with filters. pgrep -u www-data -l

Best for: Complex filtered lookups

Comparison Table

ToolTypeInteractiveInstall RequiredBest For
psSnapshotNoBuilt-inScripts & piping
topLiveYesBuilt-inQuick resource check
htopLiveYesYesDaily monitoring
pidofLookupNoBuilt-in"Is X running?"
pgrepSearchNoBuilt-inComplex filtered search

Key Takeaway

Use ps for scriptable snapshots, top/htop for interactive monitoring, and pidof/pgrep for targeted process lookup. These are the foundational tools that every higher-level monitoring system (Prometheus, Grafana) is built upon.

Lesson 4

Job Control

bg, fg, nohup — Multitasking in the terminal.

Scenario: You SSH (Secure Shell) into a remote server and start a 4-hour data migration. Do you want your terminal locked the entire time? Job control solves this.

bg — Background

When a process is stopped (Ctrl+Z), bg resumes it in the background. Your terminal prompt returns immediately.

fg — Foreground

Brings a background or stopped job back to the foreground. Use fg %1 to specify job number. Use jobs to list all jobs.

nohup — No Hang Up

Makes a process immune to SIGHUP (Signal Hang Up). When you close SSH, the process survives. Output goes to nohup.out.

Modern alternatives: tmux, screen

Job Control Workflow

State Transitions
FOREGROUND ──Ctrl+Z──► STOPPED ──bg──► BACKGROUND ▲ │ │ │ │ fg │ fg │ ▼ │ └────────────────────┘◄────────────────┘ │ nohup ▼ DETACHED (survives terminal close, output → nohup.out)
Bash
#!/usr/bin/env bash
# Simulate a long migration with job control

simulate_migration() {
    for i in $(seq 1 10); do
        echo "[$1] Batch $i of 10..."
        sleep 2
    done
    echo "[$1] Complete!"
}

# Launch in background with &
simulate_migration "Job-A" &
echo "Launched Job-A, PID: $!"

# List active jobs
jobs -l

# Launch with nohup (survives terminal close)
nohup bash -c 'sleep 60 && echo done' </dev/null &
echo "nohup job PID: $!"

# Wait for all background jobs
wait
CommandPurposeExample
bgResume stopped job in backgroundbg %1
fgBring job to foregroundfg %1
jobsList all shell jobsjobs -l
nohupImmune to SIGHUPnohup cmd &
Ctrl+ZSend SIGTSTP to foreground(keyboard)
&Launch in backgroundcmd &
Key Takeaway

Ctrl+Z stops, bg resumes in background, fg brings back, nohup survives disconnection. Master these four moves and you can multitask on any remote server. For persistent sessions, use tmux or screen.

Lesson 5

Systemd Architecture

Units, Targets, Dependencies — and the SysV init comparison.

A Brief History: SysV Init → Systemd

SysV Init — The Old Way

  • Sequential shell scripts in /etc/init.d/
  • Numbered runlevels (0–6)
  • Slow boot: 50 services started one at a time
  • No dependency management
  • No process supervision (crash = stay crashed)
  • Scattered logging

Systemd — The Modern Way

  • Parallel startup — simultaneous where dependencies allow
  • Declarative unit files — INI (Initialization)-style config
  • Automatic dependency resolution
  • Built-in supervision — auto-restart on failure
  • Unified logging — journalctl
  • Socket activation, timers, path watchers

Systemd Units — The Building Blocks

Unit TypeExtensionPurposeExample
Service.serviceA daemon or one-shot processnginx.service
Target.targetGroup of units (checkpoint)multi-user.target
Socket.socketIPC (Inter-Process Communication) / network socketsshd.socket
Timer.timerScheduled task (replaces cron)backup.timer
Mount.mountFilesystem mount pointhome.mount
Path.pathFile/directory change watcherdeploy.path

Targets Replace Runlevels

SysV RunlevelSystemd TargetDescription
0poweroff.targetShut down
1rescue.targetSingle-user / rescue mode
3multi-user.targetMulti-user, text-only (no GUI (Graphical User Interface))
5graphical.targetMulti-user with desktop GUI
6reboot.targetReboot

Dependencies — Declaring What Needs What

DirectiveTypeBehavior
Requires=Hard dependencyIf the required unit fails, this unit fails too
Wants=Soft dependencyIf the wanted unit fails, this unit still starts
After= / Before=Ordering onlyControls sequence, does NOT cause a unit to start
WantedBy=Install hookWhich target pulls in this unit at boot

Anatomy of a Unit File

[Unit] — WHO am I?

Description=
After=network.target
Requires=postgresql.service
Wants=redis.service

[Service] — HOW do I run?

Type=simple
ExecStart=/usr/bin/python3 server.py
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure

[Install] — WHEN at boot?

WantedBy=multi-user.target

Running systemctl enable creates a symlink so the service starts at boot.

Systemd vs SysV — Command Cheat Sheet

Systemd CommandSysV Init Equivalent
systemctl start myappservice myapp start
systemctl stop myappservice myapp stop
systemctl restart myappservice myapp restart
systemctl status myappservice myapp status
systemctl enable myappupdate-rc.d myapp defaults
systemctl daemon-reload(no equivalent)
journalctl -u myappcat /var/log/myapp.log

Boot Speed: Sequential vs Parallel

SysV Init (~12 seconds) — Sequential
S01 network ──► S02 database ──► S03 webapp ──► S04 cron (each waits for the previous one to finish)
Systemd (~4 seconds) — Parallel
network.target ──┬──► database.service └──► webapp.service (simultaneously!) (independent services start at the same time)

Task: Creates a systemd unit file, installs it, demonstrates enable/start/status/reload/stop, and shows the SysV command equivalents. Requires root.

Bash
#!/usr/bin/env bash
# Create and manage a systemd service (run with sudo)

# Create a unit file
cat > /etc/systemd/system/demo.service << 'EOF'
[Unit]
Description=Demo Service
After=network.target

[Service]
Type=simple
ExecStart=/usr/bin/python3 /opt/demo/server.py
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

# Reload systemd to recognize the new unit
systemctl daemon-reload

# Enable (start at boot) and start the service
systemctl enable demo.service
systemctl start demo.service

# Check status and recent logs
systemctl status demo.service
journalctl -u demo.service -n 10

# Reload config via SIGHUP (no restart)
systemctl reload demo.service

# Stop and disable
systemctl stop demo.service
systemctl disable demo.service
Key Takeaway

Systemd replaced SysV init with parallel boot, declarative unit files, automatic dependency resolution, and built-in process supervision. Master systemctl, journalctl, and unit file anatomy ([Unit], [Service], [Install]). These patterns directly mirror container orchestration in Kubernetes and Docker.

Bonus

rg + fzf + Vim & Reverse Search

Developer productivity tools for navigating real-world codebases.

Distributed-systems engineers search through thousands of files daily. Basic grep and Ctrl+R are too slow. These tools transform your terminal.

rg — ripgrep

Line-oriented search written in Rust. 5–10x faster than grep. Respects .gitignore, skips binary files, uses parallelism.

rg "database_url" --type yaml
rg "ERROR" -C 2 (context lines)
rg "pattern" --count

fzf — Fuzzy Finder

Feed it any list — files, processes, git branches, history. Type a few characters and it instantly narrows thousands of results.

ps aux | fzf → pick a process
fzf → find a file interactively

Vim + fzf.vim

The power combo. :Rg pattern searches the project. :Files fuzzy-finds files (like Ctrl+P in VS Code). :Buffers switches buffers.

Enhanced Reverse Search (Ctrl+R)

Before fzf

Shows one matching command at a time. Press Ctrl+R repeatedly to cycle. Clunky and slow.

After fzf

Shows all matching commands simultaneously. Fuzzy search with partial fragments. Scroll, preview, select with Enter. Massive productivity boost.

Integration Workflow
YOUR TERMINAL │ ├── Ctrl+R ──► fzf ──► Fuzzy match over bash history ──► Re-run command │ ├── vim :Rg ──► rg ──► Fast regex search over project ──► Jump to code line │ └── vim :Files ──► fzf ──► Fuzzy match over files ──► Open file instantly INSTALL: sudo apt install ripgrep fzf VIM PLUGIN: junegunn/fzf.vim (via vim-plug)
Key Takeaway

rg replaces grep with speed and smart defaults. fzf turns any list into an interactive fuzzy-search menu. Together with fzf.vim, they create a search-powered command center. Install both: sudo apt install ripgrep fzf.
