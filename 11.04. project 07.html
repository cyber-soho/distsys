Overview

What you are building & the "open tools" rule

A two-node PostgreSQL cluster that any real company would recognise:

  • Node A (primary) — accepts reads and writes.
  • Node B (standby) — streams WAL (Write-Ahead Log) from the primary in real time and serves read-only queries.
  • A nightly base backup automated once via cron and once via a systemd timer — so you can compare the two approaches.
  • A rehearsed failover drill that promotes B → primary in under five minutes.
  • A runbook a classmate could follow without you in the room.
⚠️

Before you run anything — read this

Some of the techniques here are identical to what a real DBA (Database Administrator) uses during a production incident. That makes them powerful learning material and also means you must respect the boundaries of the lab.

  • Scope: only the VMs you own (Vagrant, VirtualBox, KVM, a cloud VM, or a university-provisioned VM). If a machine is not yours, you do not have permission to run these commands on it — end of sentence.
  • Not production: never connect the lab cluster to real user data, a company database, or anything internet-facing that is not explicitly disposable.
  • Ethics & law: in most jurisdictions, running administrative tooling against systems you do not own is a criminal offence — see the UK's Computer Misuse Act 1990 and the EU's NIS2 Directive. These techniques exist to protect data.

⏱ ≈ 45 min

C.1 · Stop PG on Node B and quarantine its data dir

⚠️

Destructive commands ahead

You are about to delete $PGDATA on Node B. Triple-check you are logged in on B, not on A. Run hostname first.

We rename instead of delete — the recovery-hygiene habit from the lecture. rm -rf is the move that ends DBA (Database Administrator) careers. mv is reversible for five minutes, which is often enough to undo a mistake.

C.2 · Take a base backup from the primary

The password prompt will accept the one you set in B.2.

  • --pgdata → where to drop the copy.
  • --host / --username → talk to Node A as the replication user.
  • --wal-method=stream → opens a second connection to stream WAL while the base copies, so nothing is lost even if the copy is slow.
  • --write-recovery-conf → auto-generates primary_conninfo and a standby.signal file; without this you'd have to write them by hand.
  • --checkpoint=fast → do not wait for the next natural checkpoint on the primary.
  • --progress → a percentage bar so you know it did not hang.

E.4 · Re-attaching the old primary (optional stretch)

Once Node A comes back, it is not automatically part of the cluster any more. Its WAL diverged from the new timeline. Options:

  1. pg_rewind — the PG-native tool for exactly this scenario. Fast.
  2. Re-pg_basebackup from the new primary (B). Simpler, slower. Recommended for the lab.
  3. Leave A dead; rebuild from scratch next class.

If you have time, do option 2 — it is identical to Part C but swapping the roles of A and B.

E.5 · Your deliverable — the Runbook

Your runbook is a markdown file called RUNBOOK.md. Every production team has one. The single most important rule: a classmate must be able to recover from a primary crash using only your runbook, without calling you.

Minimum sections:

  1. Cluster overview. Hostnames, IPs, PG version, replication mode.
  2. Normal-operations checks. Three one-line commands to confirm everything is healthy.
  3. Backup catalogue. Where backups live, retention, how to list them, how to verify one.
  4. Restore a single table. The exact pg_restore -t command.
  5. Full restore on a new VM. The sequence from bare OS to working soholab.
  6. Failover procedure. The E.2 steps, plus any DNS / load-balancer updates.
  7. Reconcile the old primary. One paragraph on rewind vs re-base-backup with a recommendation.
  8. Escalation. Who to call at 3 a.m.
  9. Changelog. First entry is today's date + your name + "initial version".

Going further

If you finish early — or care too much

These extensions take your cluster from "lab" to "resembles production". None are required today; all will be worth points on the final exam.

  • Enable WAL archiving + point-in-time recovery (PITR). See the PostgreSQL continuous archiving docs. Rewind to any second, not just the last base backup.
  • Synchronous replication. Add synchronous_commit=on and synchronous_standby_names='pg-standby' on the primary. Measure the write-latency cost.
  • Add a third node. Cascade the WAL (A → B → C) or use quorum-based sync.
  • Automatic failover. Replace human-driven E.2 with Patroni + etcd, or repmgr, or pg_auto_failover.
  • Off-site backups. Push dumps to MinIO, AWS S3, or Backblaze B2.
  • Monitoring. In Module 10 you meet Prometheus + Grafana. Come back and instrument your cluster with postgres_exporter.
