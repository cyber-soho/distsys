Why This Matters

In July 2024, a 10-hour outage in Azure Central US caused airlines to stop checking in passengers and hospitals to revert to paper records. The root cause? Replicas disagreed on which write was the latest. This learning module answers the fundamental question of distributed systems: "Whose copy of the data do we trust — right now — when the network is slow?"

The CAP Theorem in Practice

The fundamental trilemma of distributed databases, formalized by Gilbert & Lynch (2002).

Context: This section visualizes the CAP theorem. The interactive radar chart below demonstrates how different real-world databases prioritize these three guarantees. Because network partitions (P) are inevitable in distributed systems, architects must choose to sacrifice either perfect Consistency (C) or 100% Availability (A) during an outage. Explore the buttons to see how different products make this trade-off.

C — Consistency

Every node returns the same most-recent answer.

A — Availability

Every request gets an answer (not an error).

P — Partition Tolerance

System keeps working when the network between nodes breaks. Not optional in the real world.

Eventual Consistency

"Pretty soon" is usually good enough.

Context: Eventual consistency is an engineering choice, not laziness. It allows systems to be highly available and performant. Three Flavours

  • Read-your-writes You see your own tweet immediately, others see it later.
  • Monotonic reads Once you see a value, you never see an older value.
  • Session consistency Read-your-writes within one session only.

Backup Tools & Recovery

Replication protects from hardware failure. Backup protects from the intern typing DROP TABLE.

Context: A comprehensive data safety strategy requires both logical/physical backups and a tested recovery plan. This section outlines PostgreSQL's core backup utilities and the critical steps for Point-In-Time Recovery (PITR). Explore the operational differences between tools and step through the recovery staircase to understand how to restore data to a precise moment before a catastrophic error occurred.

pg_dump

The Logical Backup

  • ✓ Connects to a running PG server, asks for data.
  • ✓ Writes a SQL script or compressed format.
  • ✓ Restores onto any compatible PG version.
When to use: Migrating versions, cloning prod to dev, backing up single schemas, databases < 500 GB.

pg_basebackup

The Physical Backup

  • ✓ Copies actual files of PG data directory byte-by-byte.
  • ✓ Server stays running during copy.
  • ✓ Restores only to same major version + architecture.
When to use: Large DBs (TB+), base of a PITR strategy, initial state of new streaming replica.

The 3-2-1 Rule (CISA Endorsed)

3 copies of the data, on 2 different media, with 1 off-site copy.

Point-In-Time Recovery Ingredients

1. Base Backup + 2. WAL Archiving Enabled + 3. Recovery Recipe

"A backup you have never restored is not a backup. It is a prayer."

The 6 Operational Steps

1

Stop PG cleanly.

2

Quarantine the broken data directory.

CRITICAL: Never rm -rf! Rename it. You may need it for forensics.

3

Extract the base backup into a fresh $PGDATA.

4

Create recovery.signal.

5

Append recovery recipe (target time) to postgresql.auto.conf.

6

Start PG → it replays WAL up to the target → promotes.
