1. Service Discovery Patterns

Service discovery is the automated "phone book" of distributed systems. It allows services to learn network locations dynamically without hard-coded IPs, solving issues caused by scaling, instances shutting down, and IP changes.

Client-Side Discovery

The calling service queries a registry directly, gets a list of healthy instances, and picks one to connect to.

2. Load Balancing Algorithms

Distributing traffic so no single machine becomes a bottleneck. Explore the algorithms below to understand their strengths and weaknesses.

Round-Robin

Hands requests to servers in a fixed rotating order (S1 → S2 → S3 → S1). Every server gets an equal share of the total traffic.

Best When: Servers are equal in power, and requests take roughly the same time to process.
Weakness: Completely ignores actual server load. A busy server will still receive new requests.

Weighted Round-Robin

More powerful servers are assigned a higher weight. A server with weight 2 receives twice as many requests as a server with weight 1.

Best When: You have a server pool with mixed hardware capacities.
Weakness: Weights are static and require manual tuning. Still ignores real-time connection state.

Least-Connections

Checks active connections on each server and sends the next request to the server with the fewest active connections. It adapts dynamically.

Best When: Requests have highly variable durations, preventing slow servers from piling up tasks.
Weakness: Requires slightly more tracking overhead at the load balancer level.

IP Hash

The client's IP address is mathematically hashed to determine the target server. Ensures the same client always hits the same server.

Best When: Session persistence (stickiness) is strictly required by the application.
Weakness: Can lead to highly uneven server loads if client IP distribution is skewed.

Random

Picks a server entirely at random. Surprisingly effective at scale due to the law of large numbers.

Best When: You have very large, homogeneous pools of servers.
Weakness: Unpredictable and highly risky with small server pools.

3. Reverse Proxies

A reverse proxy acts as the reception desk for your servers. Clients talk to the proxy; the proxy routes requests internally. Key technologies include Nginx and HAProxy.

Forward vs. Reverse Proxy

Feature Forward Proxy Reverse Proxy
Sits in front of Clients Servers
Primary Purpose Hides client identity Hides server identity
Examples VPN, Web Filters Nginx, HAProxy

Nginx (Engine-X)

  • Created 2004 (Igor Sysoev)
  • Solved C10K problem
  • Event-driven, asynchronous
  • Excels at web serving

HAProxy

  • Created 2000 (Willy Tarreau)
  • Dedicated TCP/HTTP LB
  • Extreme throughput
  • Fine-grained health checks

Nginx Config Example

upstream backend_pool {
    server 10.0.1.1:8080;
    server 10.0.1.2:8080;
    server 10.0.1.3:8080;
}

server {
    listen 80;
    location / {
        proxy_pass http://backend_pool;
    }
}

Note: Nginx defaults to Round-Robin routing across the upstream pool.

4. Health Checks & Failover

Health checks are the immune system of distributed systems. They prevent the load balancer from sending traffic to dead servers. Failover dictates how the system responds when a failure is detected.

Health Check Models

Active (Push Model)

Load balancer explicitly probes servers at intervals.

  • TCP: Network level ping
  • HTTP: GET /health surface check
  • App-Level: Deep check (DB, disk)

Passive (Pull Model)

Monitors real traffic. If a server returns many 500 errors, it gets marked unhealthy. No extra probing traffic.

Heartbeat Mechanism

Servers send "I am alive" messages to a coordinator (e.g., ZooKeeper). Missing heartbeats imply death.

Failover Flavors

Active-Passive

Active Server
Standby Server

One handles all traffic. The other sits idle waiting for failure.

Pro: Simple. Con: Wastes capacity.

Active-Active

Active Server
Active Server

Both handle traffic simultaneously. One fails, the other absorbs the share.

Pro: Efficient. Con: Requires state sync.

5. High Availability Concepts

Designing systems to remain operational for an extremely high percentage of time. HA assumes everything will eventually fail, and engineers the system to survive it.

Visualizing "The Nines"

Annual allowable downtime shrinks exponentially as availability requirements increase.

Core Strategies for High Availability

1. Redundancy

Eliminate Single Points of Failure (SPOF). Have at least two of everything (servers, DBs, power supplies).

2. Replication

Keep identical copies of data on multiple machines to survive disk failures.

3. Geographic Dist.

Spread servers across multiple data centers (multi-region) to survive localized disasters.

4. Automated Failover

Detection and recovery must be automatic. Humans are too slow for five-nines.

5. Graceful Degradation

If a subsystem fails, the rest continues with reduced functionality rather than total crashing.

6. Load Balancing

Distributes traffic to prevent bottlenecks, improving both performance and resilience.

The CAP Theorem (Briefly)

A distributed system can guarantee at most two out of three:

  • Consistency
  • Availability
  • Partition Tolerance

Because network partitions are unavoidable, systems must choose between Consistency and Availability during an outage. HA systems typically choose AP (Availability + Partition Tolerance).
