The Four Problems of Multi-Instance Services

🔍

Discovery

How does a client know which instances exist and where they live on the network?

⚖️

Distribution

How does traffic spread fairly so no single instance is overwhelmed?

🩺

Detection

How does the system know when an instance has crashed or become unhealthy?

🔄

Recovery

What happens automatically when a failure is detected?

Architecture You Will Build

CLIENT REQUEST → DNS (BIND9) → resolves "app.infrascale.local" │ ▼ HAProxy (vm-lb:80) ← Layer 7 load balancer │ │ │ round-robin / leastconn ▼ ▼ ▼ health checks every 3s Nginx:80 Nginx:80 Nginx:80 ← reverse proxy per node │ │ │ App:8080 App:8080 App:8080 ← application instances (vm-1) (vm-2) (vm-3)

Key Terminology

TermMeaning
Load Balancer (LB)Distributes incoming traffic across back-end servers.
Reverse ProxySits in front of app servers, forwarding requests. Client never talks to the app directly.
Health CheckAutomated periodic test — is the server alive and ready?
FailoverAutomatic traffic redirection from a failed server to healthy ones.
Round-RobinCycles through servers: S1 → S2 → S3 → S1…
Least-ConnRoutes to the server with fewest active connections.
TTL (Time To Live)Seconds a DNS record is cached before re-querying.

Why HAProxy + Nginx Together?

HAProxy excels at high-throughput TCP (Transmission Control Protocol) / HTTP load balancing with health checking. Nginx excels as a reverse proxy with caching, SSL (Secure Sockets Layer) / TLS (Transport Layer Security) termination, and compression. Together: HAProxy distributes, Nginx handles per-node request management.

HAProxy and Nginx are open suggestions. You may substitute with Traefik, Envoy, Caddy, or write your own. Research thoroughly and be creative.

B.2 — Deploy HAProxy as Load Balancer (30 min)

Install HAProxy on vm-lb, configure round-robin with HTTP health checks, enable the stats dashboard.

Bash
# Install HAProxy
sudo dnf install -y haproxy
haproxy -v

# Back up original, then edit
sudo cp /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg.bak
sudo nano /etc/haproxy/haproxy.cfg
HAProxy Config
# Global settings for the HAProxy process
global
    log 127.0.0.1 local2
    chroot /var/lib/haproxy
    maxconn 4000
    user haproxy
    group haproxy
    daemon

# Default timeouts and options
defaults
    mode http
    log global
    option httplog
    option dontlognull
    option http-server-close
    option forwardfor except 127.0.0.0/8
    timeout connect 10s
    timeout client 60s
    timeout server 60s
    timeout check 5s
    retries 3

# Stats dashboard on port 8404
listen stats
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 5s
    stats auth admin:infrascale

# Receive client traffic on port 80
frontend http_front
    bind *:80
    default_backend app_backend

# Pool of back-end servers with health checks
backend app_backend
    balance roundrobin
    option httpchk GET /health
    server vm-1 192.168.56.11:8080 check inter 3s fall 3 rise 2
    server vm-2 192.168.56.12:8080 check inter 3s fall 3 rise 2
    server vm-3 192.168.56.13:8080 check inter 3s fall 3 rise 2
Bash
# Validate config, then start
sudo haproxy -c -f /etc/haproxy/haproxy.cfg
sudo systemctl enable --now haproxy

# Test round-robin distribution
for i in $(seq 1 10); do
  echo -n "Request $i: "; curl -s http://192.168.56.10/health; echo
done
Stats dashboard at :8404/stats shows all 3 backends UP. Screenshot saved.

B.3 — Write Health Check Scripts (20 min)

Create a deep check script verifying HTTP, disk space, and process status. Schedule with systemd timer.

Bash — deep-health-check.sh
#!/usr/bin/env bash
set -eu
APP_URL="http://localhost:8080/health"
MIN_DISK_KB=512000
LOG="/var/log/health-check.log"
TS=$(date '+%Y-%m-%d %H:%M:%S')

# CHECK 1: HTTP response
HTTP=$(curl -sf -o /dev/null -w "%{http_code}" --max-time 5 "$APP_URL" 2>/dev/null || echo "000")
[ "$HTTP" != "200" ] && { echo "$TS FAIL: HTTP $HTTP" >> "$LOG"; exit 1; }

# CHECK 2: Disk space
AVAIL=$(df / | awk 'NR==2{print $4}')
[ "$AVAIL" -lt "$MIN_DISK_KB" ] && { echo "$TS FAIL: Disk low" >> "$LOG"; exit 1; }

# CHECK 3: Process running
if ! pgrep -x "my-web-app" >/dev/null 2>&1; then
  podman ps --filter "name=my-web-app" --filter "status=running" -q | grep -q . || \
    { echo "$TS FAIL: Process missing" >> "$LOG"; exit 1; }
fi

echo "$TS OK" >> "$LOG"; exit 0

Schedule every 30 seconds with a systemd oneshot service + timer (see full exercise doc for unit files).

journalctl -u deep-health-check.service -n 5 shows recent OK entries.

B.4 — Configure Nginx Reverse Proxy (25 min)

Install Nginx per node, proxy port 80 → app port 8080, add error pages, then update HAProxy to use port 80.

Nginx Config
upstream app_local {
    server 127.0.0.1:8080;
}
server {
    listen 80;
    server_name app.infrascale.local;

    location / {
        proxy_pass http://app_local;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }

    error_page 502 503 504 /custom_50x.html;
    location = /custom_50x.html { root /usr/share/nginx/html; internal; }
}

Then update HAProxy backends from :8080 to :80 and reload.

10-request test still rotates through all nodes, now via Nginx on port 80.

B.5 — Integrate with DNS (20 min)

Update BIND9 zones, set short TTL (Time To Live), practice adding/removing nodes dynamically.

DNS Zone

$TTL 30

        2025031101 3600 900 604800 30)

ns1     IN A   192.168.56.10
app     IN A   192.168.56.10   ; points to HAProxy

vm-2    IN A   192.168.56.12
vm-3    IN A   192.168.56.13
backend IN A   192.168.56.11   ; multi-A-record pool
backend IN A   192.168.56.12
