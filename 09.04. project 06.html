This section guides you through provisioning your OpenShift environment (Exercise 1). Before orchestrating containers, you require a functional cluster. You will choose between a cloud-based Sandbox or a local CRC installation, authenticate via the CLI, and verify your workspace. This forms the foundational layer for all subsequent tasks.

⏱️ 45 mins ⚖️ Weight: 1x

Option A — Red Hat Developer Sandbox (Recommended)

The Red Hat Developer Sandbox provides a free, shared OpenShift cluster accessible from any browser. No local installation is required.

  1. Navigate to developers.redhat.com/developer-sandbox and create a free account.
  2. Record your web console URL and API endpoint from the dashboard.
  3. Install the oc CLI on your local machine and verify: 
    oc version
  4. Log in to your sandbox cluster from the terminal: 
    oc login --token=sha256~XXXXXXXXXXXXXXXX --server=https://api.sandbox-m2.ll9k.p1.openshiftapps.com:6443
  5. Confirm connection and project access: 
    oc projects
oc project <username>-dev

Option B — CRC (Local OpenShift)

Requires at least 9 GB of free RAM and 35 GB of disk space.

crc setup
crc start
eval $(crc oc-env)
oc login -u developer -p developer https://api.crc.testing:6443

This section combines Exercises 2 and 3 into a comprehensive Workload Deployment phase. You will translate your Module 04 Docker Compose definitions into Kubernetes-native YAML manifests (Deployments, Services, ConfigMaps, Secrets). You will also implement production-grade configurations, including liveness/readiness probes and resource quotas, ensuring your deployed applications are resilient and manageable within OpenShift.

⏱️ 135 mins (Ex 2+3) ⚖️ Weight: 4x

1. Inventory & Images

Review your Compose file and tag/push your images to a registry accessible by OpenShift.

podman login docker.io -u <username>
podman tag localhost/api:v1 docker.io/<username>/infrascale-api:v1
podman push docker.io/<username>/infrascale-api:v1
oc new-project infrascale
mkdir -p ~/infrascale/openshift-manifests

2. Database Manifests (Secret, Deployment, Service)

Create the fundamental data layer. Notice the inclusion of resource limits and TCP health probes.

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  POSTGRES_PASSWORD: Y2hhbmdlLW1lLWluLXByb2R1Y3Rpb24=
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: database
spec:
  replicas: 1
  selector: { matchLabels: { component: database } }
  template:
    metadata: { labels: { app: infrascale, component: database } }
    spec:
      containers:
        - name: postgres
          image: postgres:16
          ports: [{ containerPort: 5432 }]
          envFrom: [{ secretRef: { name: db-credentials } }]
          resources:
            requests: { memory: "128Mi", cpu: "100m" }
            limits: { memory: "512Mi", cpu: "500m" }
          livenessProbe:
            tcpSocket: { port: 5432 }
            initialDelaySeconds: 20
          readinessProbe:
            exec: { command: ["pg_isready", "-U", "infrascale"] }
---
apiVersion: v1
kind: Service
metadata:
  name: database
spec:
  type: ClusterIP
  selector: { component: database }
  ports: [{ port: 5432, targetPort: 5432 }]

3. API & Frontend Manifests

Apply the same logic for your API and Frontend, utilizing HTTP probes for health checks and ConfigMaps for environmental URLs.

This section focuses on Networking and Exposure (Exercise 4). Your workloads are running but isolated. You will create OpenShift Routes to securely expose your frontend to the public internet using TLS edge termination, while understanding how internal cluster DNS resolves service names.

⏱️ 30 mins ⚖️ Weight: 2x

Expose Services to External Traffic

The oc expose command is a rapid way to generate Route resources. We will also explicitly define a secure route utilizing TLS.

oc expose service/api-backend
oc get route api-backend
curl http://<generated-api-hostname>/health

Secure Frontend with TLS Edge Termination

apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: web-frontend-secure
spec:
  to:
    kind: Service
    name: web-frontend
  port:
    targetPort: 8080
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect

This final operational section (Exercise 5) builds your fluency with the OpenShift Command-Line Interface (CLI). You will learn how to inspect resources, view logs, and execute commands directly within running containers, essential skills for debugging and maintaining production clusters.

⏱️ 30 mins ⚖️ Weight: 1x

Inspect & Describe

oc get all
oc get pods -o wide
oc get deployment/api-backend -o yaml
oc describe pod/<pod-name>

Logs & Execution

oc logs pod/<pod-name>
oc logs -f pod/<pod-name>
oc exec -it pod/<pod-name> -- /bin/sh
oc rsh pod/<pod-name>

Advanced Operations

oc get events --sort-by='.lastTimestamp'
oc port-forward service/database 5432:5432
oc scale deployment/api-backend --replicas=3
