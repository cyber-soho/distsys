Part 8

Read this before you touch anything

You have spent twelve weeks building InfraScale. Two networked Linux VMs (Virtual Machines). BIND9 DNS (Domain Name System) and OpenLDAP directory services. Hardened users, firewall rules, time synchronisation. Containers in Podman, then orchestration on OpenShift. Distributed file systems, NFS (Network File System), HDFS (Hadoop Distributed File System). Streaming replication on PostgreSQL. Load balancing with HAProxy and Nginx.

That is, frankly, a lot. More than most second-year undergraduates anywhere in Europe. You should be a little bit proud.

There is one giant, glaring problem with everything you have built so far: you cannot see what is happening inside it. Right now, if your checkout service starts answering 30% slower at 03:14 a.m., you will find out about it from an angry email — assuming somebody is awake to send one. That has to change.

"How do you know what is happening inside a distributed system right now?"

By the end of this Project Task, you will be able to say, with confidence, what your platform is doing, why it is doing it, and how to put a fix into production without humans typing commands at 4 a.m.

Two things, really. Observability — making the system tell you what is happening on its own. Metrics, logs, dashboards, alerts. CI/CD (Continuous Integration / Continuous Delivery) — making changes flow into production through a pipeline that tests them, builds an image, pushes it to a registry, and deploys it. The two halves are the same idea: automation in service of humans who would like to sleep at night.

The deliverable, per the syllabus: complete observability stack + automated CI/CD pipeline + Ansible playbooks. Block out half a Saturday — six hours, a strong coffee, and good wifi — and you will finish. If you only have four hours, prioritise Parts 1–6; Parts 7–8 can spill into the next session.

⚠ Disclaimers — please read them, they exist for good reasons

Three things you must understand before you touch any tool below.

⚠ Disclaimer 1 · Teaching environment, not a real one

Same tools as production. Same risks.

Some of the tools you will install — Prometheus, Grafana, Loki, Ansible, Tekton, OpenShift Pipelines — are literally the same tools that run in production at Spotify, Booking.com, the Spanish INE, most banks, and most large universities. They are not toy versions. They are the real thing.

  • Run everything inside your own InfraScale lab environment. Local VMs. CodeReady Containers. Red Hat Developer Sandbox. Never point any of these tools at infrastructure you do not personally own and have explicit written authorisation to operate.
  • Use only fake credentials. Every API key, password, token, webhook URL, and SSH key here must be a throw-away test value with no access to anything real.
  • Document every step. Your runbook is the only thing standing between "I learned a lot" and "I broke a thing".
▸ Disclaimer 2 · Every tool below is OPEN

Replace it. Improve it. Build a better one.

Every piece of software named in this lab is one example of a class of tools, not the only correct answer. The shopping list says "Prometheus + Grafana", but a serious team weighs alternatives:

  • Metrics: VictoriaMetrics, Mimir, InfluxDB, Thanos.
  • Dashboards: Kibana, Superset, hand-rolled in Jupyter.
  • CI/CD: GitLab CI, Jenkins, Argo Workflows, GitHub Actions, Drone.
  • Provisioning: Salt, Puppet, Chef, Pulumi, Terraform.
  • Logs: ELK stack, classical rsyslog, Graylog, Vector, Fluentd.

Research at least one alternative for every tool you use today. Write three honest sentences in your final report on why you picked what you picked, or why you would pick differently. Critical thinking matters more than tool fluency. If you are brave: build something yourself.

⚠ Disclaimer 3 · Verify links before you click

Three seconds. Saves careers.

Every URL in this document was correct on the day this Project Task was written. URLs decay. Domains expire. Accounts get compromised. Before clicking any link, paste it into the VirusTotal URL scanner and confirm it is clean.

This is a habit you should build now and keep for life. It costs three seconds and has saved more careers than I can count.

Part One ≈ 45 min weight ×1

Deploy Prometheus + Grafana on OpenShift

What we are doing and why

OpenShift ships with its own cluster monitoring stack — a dedicated Prometheus that watches the cluster's own health (etcd, the API server, the kubelet, etc.). That instance is not the one we want. It is locked down, opinionated, and not designed to be customised. Touching it is a fast way to break your cluster.

Instead, we install a second Prometheus + Grafana specifically for our application metrics. This is a normal pattern in real production OpenShift deployments. The official, supported way is via the user-workload monitoring feature, or by installing the Prometheus Operator into your own namespace. We use the second route because it teaches you more about how the pieces fit together.

Step 1.1 — Create a project for your observability stack

oc new-project is the OpenShift convention; under the hood it creates a Kubernetes namespace plus some OpenShift-specific resources (ProjectRequest, RoleBindings). If you forget which project you are in, run oc project.

Step 1.2 — Install the Prometheus Operator

Operators handle upgrades, custom resources, and reconciliation for you, which is what you want for a system supposed to outlive any single human. From the OpenShift web console (Operators → OperatorHub), search for Prometheus Operator and install it into the infrascale-observability namespace.

The CLI equivalent — which I recommend, because clicking around the web console is not a skill you can put in a runbook — is:

▸ if the operator does not appear

Your cluster might not have the community-operators CatalogSource. Run oc get catalogsource -A to check. If you are on the Red Hat Developer Sandbox, some operator subscriptions are restricted; fall back to a plain Helm install of the kube-prometheus-stack chart. Both routes lead to the same place.

Step 1.3 — Define a Prometheus instance

Two important details, because you should never copy YAML you do not understand:

  • serviceMonitorSelector: {} (empty selector) means select all ServiceMonitors regardless of labels. In a real cluster shared with other teams, you would put a label here to scope it.
  • retention: 7d means Prometheus keeps a week of metrics. For long-term storage, you would add a remote-write target like Mimir or Thanos. Not today.

You will also need a ServiceAccount, Role, and RoleBinding for the prometheus SA — the Prometheus Operator quickstart has a minimal RBAC example you can copy. Adapt the namespace.

Step 1.4 — Expose Prometheus via a Route

Open the URL in a browser. You should see Prometheus's bare web UI. Try a query: type up and press Execute. Empty? Good — we have not added any targets yet. We fix that in Parts 2 and 3.

Step 1.5 — Install Grafana

The cleanest route is the Grafana Operator, but for a four-hour lab the official Grafana container driven by a plain Deployment is faster.

Open Grafana. Log in with admin / ChangeThisInTheLab!. Change the password immediately, even though this is a lab. Build the habit now.

▸ Sit with this idea

The YAML you just wrote is declarative. You said "I want a Deployment that looks like this" and the cluster reconciled toward that state. Compare that to a Bash script that imperatively runs podman run .... If a node dies, the imperative script doesn't know — it already finished. The declarative object knows, because the cluster keeps comparing real state to desired state forever. This is the single biggest mental shift between sysadmin work and platform engineering.

Step 1.6 — Wire Grafana to Prometheus

In Grafana → Connections → Data sources → Add data source → Prometheus. URL: http://prometheus-operated.infrascale-observability.svc:9090. Save & test. Green tick? You are wired.

▸ End of Part 1 deliverable

Prometheus and Grafana, both running, both exposed via Routes, with Grafana able to read from Prometheus. Take a screenshot of the green "Data source is working" message and save it as module10/docs/grafana-datasource-ok.png.

Part Two ≈ 30 min weight ×1

Configure node_exporter on every node

What we are doing and why

node_exporter is a tiny Go binary written by the Prometheus authors that exposes Linux machine-level metrics — CPU (Central Processing Unit), memory, disk, network, filesystem — on a /metrics HTTP (HyperText Transfer Protocol) endpoint, in the standard Prometheus exposition format. Every Linux box that wants to be visible to Prometheus needs something exporting metrics, and node_exporter is the de-facto standard for general-purpose host metrics.

In an OpenShift cluster, a node_exporter already runs on every cluster node as part of the cluster monitoring stack — but it's behind RBAC (Role-Based Access Control) you can't easily reach. So we deploy our own node_exporter on the lab VMs that sit outside the cluster (your DNS server, your LDAP server, your PostgreSQL primary and standby — whichever VMs from Modules 1–9 are still running).

Step 2.1 — Install node_exporter on each VM

The fastest route, on RHEL/CentOS Stream/Fedora, is via the EPEL (Extra Packages for Enterprise Linux) repository:

If your distribution does not package it, download the latest release tarball from the Prometheus releases page, extract, copy the binary to /usr/local/bin/, and write a node_exporter.service systemd unit. Both routes are valid. The package route is faster; the binary route teaches you more.

By default, node_exporter listens on :9100. Confirm:

You should see a torrent of lines like node_cpu_seconds_total{...}, node_memory_MemAvailable_bytes, node_filesystem_size_bytes. That is the Prometheus exposition format. Read it. Recognise the shape. This is the raw material of every dashboard you will ever build.

Step 2.2 — Open the firewall

From Module 3 you should already have firewalld running on each VM. Allow Prometheus to reach the new port:

⚠ real-world warning

9100/tcp is exposed to the trusted zone in this lab. In production, you would scope it to a specific source IP range — the IP of your Prometheus server — using firewall-cmd --add-rich-rule. We are not doing that today, but flag it in your report as a "next step before this is production-ready".

Step 2.3 — Tell Prometheus about the new targets

Because Prometheus runs inside OpenShift and the targets are outside the cluster, we cannot use a ServiceMonitor (those only discover services inside the cluster). We need a plain scrape config, mounted as a Secret and referenced from the Prometheus resource. The Prometheus Operator documents this pattern under additionalScrapeConfigs.

Wait ~30 seconds, then go back to the Prometheus web UI → Status → Targets. You should see your VMs as UP. If DOWN, common causes: DNS does not resolve from inside the cluster (test with oc debug node/...), the firewall did not actually reload (sudo firewall-cmd --list-all), or node_exporter is binding to 127.0.0.1 only (check --web.listen-address in the systemd unit).

▸ End of Part 2 deliverable

All your InfraScale lab VMs visible as UP targets in Prometheus. Screenshot module10/docs/prometheus-targets-up.png.

Part Four ≈ 30 min weight ×2

Centralised logging — rsyslog OR Loki

What we are doing and why

Metrics tell you that something is wrong. Logs tell you why. Both are necessary. You will pick one of two paths in this part — both are valid, both are widely used in industry, and both work for InfraScale.

Path A — rsyslog (the classical, every-Linux-box approach)

The flow: every VM's local rsyslog forwards logs over TCP (Transmission Control Protocol) port 514 (or, ideally, RELP — Reliable Event Logging Protocol — for guaranteed delivery) to a central rsyslog collector VM. Queries are done with grep, awk, journalctl, or GoAccess. It is boring, bulletproof, and what every bank in Europe still runs.

The official rsyslog documentation is excellent and has more detail than I can reproduce here.

Path B — Loki + Promtail (the cloud-native, query-as-code approach)

The flow: each node runs Promtail (a tiny agent, like node_exporter but for logs), which reads /var/log/* and ships entries to a central Loki instance running on OpenShift. Loki indexes only metadata (labels), not the log content itself, which makes it cheap and fast. Queries happen in Grafana with LogQL, which looks suspiciously like PromQL on purpose.

3. In Grafana → Connections → Add data source → Loki. URL http://loki.infrascale-observability.svc:3100. Save & test.

4. In Grafana → Explore → pick the Loki data source. Try {job="varlogs"} and select a host. You are reading your VM logs in a browser. That is exactly what was missing from your platform.

▸ why two paths

In the real world, a bank with a 2008-vintage SIEM (Security Information and Event Management) gets rsyslog. A startup with an OpenShift-shaped budget gets Loki. You will encounter both. Pick whichever interests you more for this lab. If you are ambitious, do both, and write 200 words in your report comparing them.

Part Five ≈ 45 min weight ×2

Build dashboards and configure alerts

What we are doing and why

Numbers in Prometheus and lines in Loki are raw material. The deliverable that an on-call engineer actually uses is a dashboard a human can scan in five seconds plus an alert that wakes them up only when they should be woken up.

Step 5.1 — One real dashboard

In Grafana → Dashboards → New. Add four panels for your web-app service, the famous RED framework from Tom Wilkie's 2017 talk at Monitorama (Rate, Errors, Duration). One panel per:

  • Rate — sum(rate(http_requests_total[1m])) (requests per second).
  • Errors — sum(rate(http_requests_total{status=~"5.."}[1m])) (5xx per second).
  • Duration p99 — histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le)).
  • Up — up{job="..."} displayed as a single stat with a green/red threshold.

Lay them out in a 2 × 2 grid. Add a meaningful dashboard title ("InfraScale — web-app — production"). Save. Screenshot.

▸ a philosophy break

The instinct of every junior engineer is to add more panels. Resist it. A dashboard with 47 panels gets ignored. A dashboard with 4 panels, each answering a real question, gets opened during incidents. Less is more. Google's SRE Book — the reference here — has a whole chapter on this. Read it tonight if you have time.

Part Six ≈ 45 min weight ×2

Ansible playbooks for infrastructure provisioning

What we are doing and why

Up to now you have been running commands by hand. That works for learning. It does not work for operating. The two tests that distinguish a runbook from a reproducible deployment are:

  1. Can a different person, who has never touched your VMs, recreate the entire setup from scratch on fresh machines?
  2. Can the same person do it again next week without remembering anything?

Ansible answers both. A playbook is a YAML file that says "this is the desired state of these machines". You run it. Ansible figures out what to change. It is idempotent — running it twice has the same effect as running it once. Re-runs are cheap. That property is the whole point.

Step 6.2 — A playbook to install node_exporter on every VM

Notice what just happened. Two minutes ago you had to SSH into four VMs and run dnf install four times. Now you have a 25-line file that does it idempotently, and that lives in Git, where it can be reviewed in a Pull Request and audited later. That is the whole shift Module 10 is about.

Step 6.3 — A second playbook for Promtail (or rsyslog)

Write a playbook that installs and configures Promtail on every VM (Path B from Part 4) or that configures forward rules on every VM (Path A from Part 4). The exact tasks depend on which path you took. Aim for fewer than 60 lines. Idempotent. Re-runnable.

▸ stretch

Put the Promtail config in templates/promtail.yml.j2 with Jinja2 placeholders for hostname, then use the ansible.builtin.template module to render it. This is how every real Ansible playbook in industry works.

Part Seven ≈ 45 min weight ×2

Tekton pipeline: test → build → push → deploy

What we are doing and why

Up to this point, every container-image change you have made was: edit code, podman build, podman push, oc set image. Four manual steps. Easy to do wrong at 3 a.m. Easy to skip the test step "just this once". Easy to push the wrong tag. We are going to fix all of that.

A CI/CD pipeline is a graph of tasks, defined as code, triggered by a Git push, that performs those four steps in a deterministic order. We use OpenShift Pipelines, which is Red Hat's distribution of Tekton, the CNCF (Cloud Native Computing Foundation) project for Kubernetes-native CI/CD.

Step 7.1 — Install OpenShift Pipelines

In a real cluster a cluster-admin installs the operator from OperatorHub. On the Developer Sandbox, it is preinstalled. Confirm with:

If tkn is missing, download the binary and put it on $PATH.

Step 7.2 — Author Pipeline-as-Code

Tekton pipelines are made of Tasks, Pipelines, and PipelineRuns. The reusable Tasks for git-clone, container build (with Buildah), and oc apply already exist in the Tekton Hub. You install them once into your project; you reuse them everywhere.

Watch the four tasks turn green one by one in the OpenShift Pipelines view. The first time you see this is genuinely satisfying.

Step 7.3 — Trigger on Git push

For full GitOps flow, attach a Tekton Trigger that listens to a webhook from your Git provider. The pattern is well documented; aim to wire one up if you have time.

Part Eight ≈ 30 min weight ×1

Rolling deployments + full documentation

What we are doing and why

OpenShift's default deployment strategy is RollingUpdate. You did not have to do anything in Part 7 to get rolling deployments — oc set image triggered one for free. Your job in this part is to prove it works and document the entire system.

Step 8.1 — Watch a rolling deployment happen

Open three terminal panes:

Watch the magic. New pods come up. Old pods drain. Not a single request returns 503, because the readiness probe holds traffic until each new pod is genuinely ready. If any request did fail, your readiness probe is wrong; this is the moment to fix it. Add an X-App-Version HTTP header to your service so you can literally see the version flip mid-loop.

Step 8.2 — Configure deployment strategy parameters

Apply, run the pipeline again, observe the difference. These two numbers are what every SRE (Site Reliability Engineer) on Earth has thought about at some point in their career. Tune them for your tolerance for risk vs. speed.

Step 8.3 — Document everything

Create module10/docs/RUNBOOK.md. It must contain, at minimum:

  • Architecture diagram of the observability stack (ASCII is fine; even a hand-drawn photo is fine if it's legible).
  • How to bootstrap from zero: run these Ansible playbooks in this order, then apply these YAMLs in this order.
  • How to roll out a new version: git push followed by one CLI command.
  • How to roll back: oc rollout undo deployment/web-app.
  • How to silence an alert during planned maintenance: open Alertmanager UI, create a silence with the right matchers, set duration.
  • Three things you would do differently if you were rebuilding this for production. (TLS — Transport Layer Security — between Promtail and Loki. mTLS — mutual TLS — between Prometheus and node_exporter. A secrets manager rather than YAML literal passwords. Etc.)
A runbook that exists only in someone's head is not a runbook. It is a hostage situation.

  • Run end-to-end on a fresh VM. Confirm idempotency: second run = zero changes.

Part 7 · Tekton pipeline test → build → push → deploy (45 min · ×2)

  • Confirm OpenShift Pipelines installed: tkn version.
  • Author pipeline.yaml with four sequential tasks: git-clone, pytest, Buildah build, openshift-client deploy.
  • Trigger via tkn pipeline start ... --showlog. All four tasks green.
  • Stretch: wire a Tekton Trigger to a Git webhook.

Part 8 · Rolling deployments + RUNBOOK (30 min · ×1)

  • Set strategy.rollingUpdate.maxUnavailable: 25%, maxSurge: 25%.
  • Trigger a pipeline with a new tag while running pod-watch + curl-loop. Verify zero 5xx.
  • Practise rollback: oc rollout undo deployment/web-app. Time it.
  • Write RUNBOOK.md: architecture, bootstrap order, deploy/rollback/silence procedures, 200-word reflection.
