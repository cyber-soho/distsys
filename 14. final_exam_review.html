Chapter 1 — Container Orchestration with OpenShift

Chapter 1

Why container orchestration exists at all

A container is a lightweight package that bundles application code together with every library and configuration setting it needs to run. Running one or two containers on a developer laptop is trivial. Running ten thousand containers across two cloud providers and three data centres, with one of them dying every few minutes for reasons nobody quite remembers, is the problem that container orchestration was invented to solve. According to the Cloud Native Computing Foundation (CNCF), the great majority of organisations now run containers in production, and at that scale, no team of humans can keep up with the work by hand.

A good orchestrator handles, automatically and continuously, every operational concern that a human operator would otherwise have to deal with at three in the morning: scheduling and placement, scaling, self-healing, load balancing, rolling updates and rollbacks, service discovery, and the secure injection of configuration and secrets. The orchestrator is the conductor of the orchestra; the individual containers are the musicians.

The major orchestration platforms you will meet in industry are Kubernetes (K8s), an open-source project maintained by CNCF; Red Hat OpenShift, Red Hat's enterprise distribution built on top of Kubernetes; Docker Swarm, the simpler, smaller-deployment cousin from Docker, Inc.; and managed cloud services such as Amazon Elastic Container Service (Amazon ECS) on AWS (Amazon Web Services).

OpenShift architecture — Master, Worker, and etcd

Every OpenShift cluster is built from three fundamental pieces. Picture a well-run hospital and the analogy will stick.

Master nodes (the control plane) are the brains of the cluster. They do not run your application workloads; instead they manage, schedule, and monitor everything that happens across the cluster. A production deployment runs at least three master nodes for High Availability (HA). The services that live on a master node include the kube-apiserver (the single front door for all cluster operations — every command from a human, a script, or another component passes through it first); the kube-scheduler (which decides which worker node should run each new container based on available CPU — Central Processing Unit — and memory); the kube-controller-manager (a collection of control loops that continuously compare the desired state with the actual state and take corrective action); the OpenShift API server (which extends the Kubernetes API with OpenShift-specific objects such as Routes and BuildConfigs); and the OpenShift OAuth server (which handles user authentication so that developers and administrators can log in securely).

Worker nodes (the compute plane) are where your actual application containers live and run. Each worker runs kubelet, the agent that receives instructions from the master; CRI-O (Container Runtime Interface — Open), the container engine that actually starts and stops container processes (OpenShift uses CRI-O instead of Docker); and kube-proxy, which manages the network rules that get traffic to the right container.

etcd is the cluster's medical-records system — an open-source, strongly consistent, distributed key-value store that holds the single source of truth for every object in the cluster. Every pod, every service, every configuration setting is persisted inside etcd. Lose etcd and the cluster loses its memory. Two properties of etcd matter for the exam: it uses the Raft consensus algorithm to keep its data consistent across an odd number of members (three or five in production, so a quorum can always be formed), and it should run on Solid-State Drive (SSD) storage because slow disks are the single most common cause of etcd instability.

The operating system underneath every OpenShift 4.x node is Red Hat Enterprise Linux CoreOS (RHCOS). RHCOS is minimal, immutable, container-optimised, and managed declaratively by the Machine Config Operator (MCO), which means the operating system itself is treated as code.

The flow of one request through the cluster

A developer submits a request — "run three copies of my web app" — through the oc CLI (OpenShift Command-Line Interface) or the web console. The request hits the API server on a master node. The API server validates the request, authenticates the user via the OAuth server, and writes the desired state to etcd. The scheduler notices the new pods have not been assigned to any node and selects appropriate worker nodes based on available resources. The kubelet on each chosen worker pulls the container image and starts the container via CRI-O. The controller manager continuously monitors the actual state; if a container later crashes, it instructs the scheduler to place a replacement. The developer never has to be involved in any of this.

Projects, Pods, Deployments, Services, Routes

OpenShift organises your work into a clear hierarchy. A Project is a namespaced workspace that isolates resources, applies quotas, and controls who can access what — a Project is a Kubernetes Namespace with some additional administrative features such as default network policies. A Pod is the smallest deployable unit — one or more tightly coupled containers sharing the same network and storage; most pods contain just one container. A Deployment is a declarative description of your desired state: which container image to run, how many replicas, and how to update them; the Deployment controller is responsible for ensuring reality matches your declaration. A Service is a stable internal network endpoint that load-balances traffic across all pods matching a label selector — pods come and go, but the Service's internal address (ClusterIP) stays constant. A Route is an OpenShift-specific resource that exposes a Service to the outside world via a human-readable hostname, with optional Transport Layer Security (TLS) termination.

Together they form a clean chain: the Internet talks to a Route (public Uniform Resource Locator), which forwards traffic to a Service (internal load balancer), which forwards traffic to one or more Pods (running containers), which are managed by a Deployment (replica controller), which lives inside a Project (isolation boundary).

The oc command-line interface

The oc tool is OpenShift's primary Command-Line Interface (CLI). It is a superset of kubectl — every kubectl command works inside oc, but oc adds extras for OpenShift-specific resources. Commands every student should be able to recognise on sight include oc login https://api.cluster.example.com to authenticate, oc new-project my-app to create a new project, oc new-app nginx:latest to deploy an application from an image, oc get pods to list pods, oc get routes to list external URLs, oc expose svc/my-service to create a Route exposing a Service, oc scale deployment/my-app --replicas=5 to scale a Deployment, oc logs pod/my-app-abc12 to view a pod's container logs, oc describe pod/my-app-abc12 to get verbose detail, and oc delete project my-app to delete a Project together with everything it contains.

OpenShift versus Kubernetes — the honest comparison

OpenShift is Kubernetes. Every OpenShift cluster runs a complete, certified Kubernetes distribution underneath. The kubectl command works on OpenShift; standard Kubernetes YAML (YAML Ain't Markup Language) manifests deploy on OpenShift without modification. OpenShift does not compete with Kubernetes — it builds on top of it. The clean mental model is Kubernetes is to Linux as OpenShift is to Red Hat Enterprise Linux: both run the same kernel, but one adds enterprise packaging, security hardening, a commercial support contract, and a curated set of pre-integrated tools.

The major differences in practice: OpenShift's defaults are stricter (Security Context Constraints — SCCs — restrict container privileges, and many Docker Hub images will not run without explicit permission); the web console is much richer (with separate Developer and Administrator perspectives); the CI/CD pipeline engine (OpenShift Pipelines, based on Tekton) and the container image registry are built in rather than bolted on; OpenShift uses Routes instead of standard Kubernetes Ingress for external traffic; OpenShift ships with the Cluster Version Operator (CVO) that handles over-the-air updates; and OpenShift requires RHCOS on the control plane nodes rather than letting you bring your own Linux distribution.

Most large enterprises use both. Vanilla Kubernetes for development and experimentation; OpenShift for production workloads where security, support, and operational consistency are non-negotiable. The skills transfer cleanly because OpenShift is Kubernetes at its core.

Chapter 2 — Distributed File Systems

Chapter 2

Storage, from the physical bottom up

Storage devices break down into three useful axes: the physical medium, the connection architecture, and the access semantics.

By physical medium, the canon is: the HDD (Hard Disk Drive), with a magnetic platter rotating between 5,400 and 15,000 Revolutions Per Minute (RPM) — cheap per gigabyte, mechanically slow; the SSD (Solid State Drive), built from NAND (Negated AND) flash memory cells with no moving parts — much faster, more expensive per gigabyte, with the gap closing every year; NVMe (Non-Volatile Memory Express), an interface and protocol designed specifically for SSDs that rides the PCIe (Peripheral Component Interconnect Express) bus instead of the older SATA (Serial Advanced Technology Attachment) bus — dramatically higher bandwidth and lower latency; tape storage, the oldest surviving medium, with modern cartridges from the LTO (Linear Tape-Open) programme holding 18 Terabytes (TB) uncompressed and surviving thirty years, used for archives by organisations like CERN; and optical media (Compact Disc, Digital Versatile Disc, Blu-ray), now largely legacy outside specialised archival workflows.

By connection architecture, four categories matter for distributed systems. DAS (Direct-Attached Storage) is physically connected to one machine — your laptop's internal drive, a server's internal RAID (Redundant Array of Independent Disks) array. Fast, simple, and unshareable. NAS (Network-Attached Storage) is a dedicated device on the network that provides file-level access via NFS (Network File System) or SMB/CIFS (Server Message Block / Common Internet File System); multiple users can hit it at once — like a shared kitchen refrigerator. SAN (Storage Area Network) is a high-speed, dedicated network providing block-level storage to servers, using protocols such as Fibre Channel, iSCSI (Internet Small Computer System Interface), or NVMe-oF (NVMe over Fabrics); SANs are expensive, complex, and standard in hospital records systems, stock-exchange databases, and large virtualisation clusters. Cloud storage is provided as a service over the internet by AWS (Amazon Web Services), Microsoft Azure, and Google Cloud Platform (GCP), in three flavours that map onto the categories above (block, file, object).

Object storage deserves a special note. It does not use a file hierarchy at all — no folders. Each piece of data is stored as an object with a unique identifier, metadata, and the data itself, in a flat address space. It scales practically infinitely. It is accessed via HTTP (Hypertext Transfer Protocol) / REST (Representational State Transfer) Application Programming Interfaces (APIs). Examples include AWS Simple Storage Service (S3), Azure Blob Storage, and GCP Cloud Storage.

Local file systems and journaling

The file systems you will meet most often include FAT32 (File Allocation Table, 32-bit) — universally compatible, but with a hard four-Gigabyte (GB) single-file limit; exFAT (Extended File Allocation Table) — Microsoft's upgrade for flash drives and Secure Digital Extended Capacity (SDXC) cards, removing the 4 GB limit and standardised by the SD Association; NTFS (New Technology File System) — the Windows default since 1993, supporting Access Control Lists (ACLs), encryption via the Encrypting File System (EFS), and journaling; ext4 (Fourth Extended Filesystem) — the Linux workhorse, default on most distributions, supporting up to 16 TB files; XFS (X File System) — originally from Silicon Graphics, Inc. (SGI), the default on Red Hat Enterprise Linux (RHEL), excelling at large files and parallel I/O (Input/Output); Btrfs (B-Tree File System) — a modern copy-on-write Linux file system used by default in SUSE Linux Enterprise; ZFS (Zettabyte File System) — originally from Sun Microsystems (now Oracle), combining file system and volume manager into one, with checksums on everything, built-in RAID-Z, snapshots, deduplication, and compression; and APFS (Apple File System) — introduced by Apple in 2017, optimised for flash, with strong encryption and nanosecond-resolution timestamps.

A journaling file system records its planned changes in a journal before committing them to the main storage area. If the machine crashes mid-write, it can replay the journal and recover to a consistent state. NTFS, ext3, ext4, XFS, HFS+ (Hierarchical File System Plus), and Btrfs all journal; ZFS uses copy-on-write with transaction groups for the same purpose. Without a journal, a power cut in the middle of a write can leave you with corrupted files and a long, sad afternoon with fsck.

A few special-purpose file systems are worth recognising: tmpfs lives entirely in RAM (Random Access Memory) and is cleared on every reboot — Linux uses it for /tmp and /run for raw speed; procfs and sysfs are virtual file systems that expose kernel and process information as files (running cat /proc/cpuinfo reads from procfs — no actual file is on disk); and swap is a designated area on disk used as overflow memory when physical RAM is full.

Partitions versus volumes

A partition is a physical division of a disk — a contiguous section of the disk treated as an independent unit, with its entry in the disk's partition table at the very beginning of the disk. The two partition-table formats you will see are MBR (Master Boot Record), the legacy scheme from the 1983 IBM PC era, with a maximum of four primary partitions per disk and a hard 2 TB disk-size limit; and GPT (GUID Partition Table), the modern replacement that ships as part of the UEFI (Unified Extensible Firmware Interface) specification, supporting up to 128 partitions by default and addressing disks up to 9.4 Zettabytes (ZB).

A volume is a logical storage unit — formatted with a file system and mountable by the operating system. In many cases one partition equals one volume, but the two are not synonymous: a volume can span multiple partitions or multiple disks (via the Linux Logical Volume Manager — LVM — or software RAID), one partition can hold multiple subvolumes (in ZFS or Btrfs), and a volume can exist without any traditional partition at all (a RAID array presented as a single logical volume).

The volume types you should be able to name: Simple Volume (one volume on one partition); Spanned Volume (one volume across multiple disks); Striped Volume / RAID 0 (data split across disks in parallel for speed — no redundancy); Mirrored Volume / RAID 1 (data duplicated identically across two or more disks); RAID-5 Volume (data plus parity distributed across three or more disks, surviving one disk failure); and LVM Logical Volume, built on top of Physical Volumes (PVs), grouped into Volume Groups (VGs), from which Logical Volumes (LVs) are carved — the magic of LVM is that you can resize, snapshot, and move volumes without downtime.

What a distributed file system actually is

A Distributed File System (DFS) stores data across multiple physical machines connected by a network, but presents the data to users and applications as if it were on a single local machine. The fundamental goal is location transparency — files appear to be local even though they are spread across a rack, a data centre, or the globe.

Every DFS, regardless of implementation, contains the same core architectural elements. A metadata server (or name server) is the brain — it knows where every file is, which chunks it is broken into, which nodes hold those chunks, and what permissions apply. It does not store the file data itself; it stores the map. Data servers (or storage nodes) are the muscles — each holds the actual file data on its own local disk and responds to read/write requests directed to it. The client is the application or user; it talks to the metadata server to find out where data lives, then communicates directly with the data servers to transfer the data. This two-step pattern keeps the metadata server from becoming a bandwidth bottleneck.

The classic DFS design challenges you will be asked about are consistency (what happens when two users edit the same file at the same time?), replication (how many copies, on which nodes?), partition tolerance (what happens when the network between two halves of the cluster goes down?), and caching (how do you avoid serving stale data?).

NFS — exports, mounts, and the things that go wrong

The NFS (Network File System) protocol, originally from Sun Microsystems in 1984, is the standard for Unix and Linux file sharing. It follows the client-server model and uses RPC (Remote Procedure Call) for communication. On the server, a directory becomes available to clients by being listed in /etc/exports; the client mounts it with the mount command and from that point on the operating system on the client intercepts standard file calls and translates them into NFS RPC calls over the network.

The versions you should know: NFSv2 (1989, RFC 1094) — stateless, User Datagram Protocol (UDP)-only, 2 GB file size limit, effectively obsolete; NFSv3 (1995, RFC 1813) — added Transmission Control Protocol (TCP) support, asynchronous writes, 64-bit file sizes, still widely deployed; NFSv4 (2003, RFC 7530) — stateful, built-in security via Kerberos, compound operations, mandatory Access Control List (ACL) support, uses only TCP port 2049 (which makes firewalls much happier).

A typical /etc/exports line reads /data/shared 192.168.1.0/24(rw,sync,no_subtree_check). The export options every student should recognise are rw (read-write), ro (read-only), sync (write completes only after data is on stable storage — safe, slower), async (write returns before data hits disk — faster, risky), no_subtree_check (improves performance, recommended), root_squash (maps client root to the anonymous user on the server — a security measure), and no_root_squash (allows client root to act as server root — use with caution).

On the client side, the mount options you should recognise are hard (keep retrying indefinitely if the server is unavailable — safe for critical data), soft (eventually report an error — prevents hangs but risks data corruption), intr (allow Ctrl+C to cancel a hung operation), timeo=N (timeout in tenths of a second), retrans=N (number of retries), vers=4 (force NFSv4), and sec=krb5 (use Kerberos for authentication). Persistent mounts that survive reboots go into /etc/fstab (the File System Table) with the file system type nfs.

HDFS — the Hadoop Distributed File System

HDFS (Hadoop Distributed File System) is the storage layer of Apache Hadoop and was directly inspired by Google's internal GFS (Google File System) paper from 2003. It uses a master-slave architecture with three key daemons.

The NameNode is the single master — the librarian. It does not store any file data; it stores metadata: which files exist, what permissions they have, which blocks make up each file, and which DataNodes hold those blocks. The metadata lives in the NameNode's RAM for speed. On disk, the NameNode keeps two files: the FsImage, a snapshot of the file system namespace at a given point in time, and the EditLog, a sequential log of every change made since the last FsImage snapshot.

DataNodes are the worker machines that hold the actual data blocks on local disks (typically ext4 or XFS). Each DataNode sends two kinds of messages to the NameNode: a heartbeat every three seconds saying "I am alive", and a block report that lists every block stored on that node so the NameNode can verify its metadata. If the NameNode has not heard a heartbeat from a DataNode in ten minutes (the default), it marks the node dead and triggers re-replication of every block that lived there.

The Secondary NameNode is not a backup. Despite the name, its job is to periodically merge the FsImage and EditLog into a fresh checkpoint so the EditLog does not grow indefinitely. The Secondary NameNode is one of the most misleadingly named components in distributed systems history.

When a client writes a file to HDFS, the file is split into fixed-size blocks — 128 Megabytes (MB) in modern Hadoop, 64 MB in older versions — and each block is replicated to multiple DataNodes (default replication factor of 3). For a 1 GB file with replication factor 3, you have 8 blocks, each with 3 replicas, for a total of 24 block instances and 3 GB of consumed storage. HDFS uses a rack-aware placement policy for those replicas: the first replica goes on the local DataNode (or a random node in the local rack), the second goes on a DataNode in a different rack, and the third on another DataNode in that same remote rack. This balances fault tolerance (the cluster survives the loss of an entire rack) with write performance (only one cross-rack network transfer is required).

Replication and fault tolerance — what actually happens when things break

Replication means keeping multiple identical copies of data on different machines, ideally on different racks or in different data centres. The Replication Factor (RF) is the number of copies. Higher RF means more safety but more storage and network cost.

Fault detection rests on two mechanisms: heartbeats (as described above) and checksums. HDFS computes a checksum for every block when it is written, and the client verifies the checksum on every read. If the checksum does not match, the block is treated as corrupted, the client transparently fetches the block from another replica, and the NameNode schedules re-replication of the healthy copy.

The categories of failure a DFS routinely handles are: a single disk failure (NameNode detects via missing heartbeats, re-replicates the affected blocks); a whole node failure (same response, more blocks to copy); a rack failure (rack-aware placement ensures replicas exist on other racks); a network partition (the NameNode cannot reach some nodes — split-brain risk if not handled carefully); a NameNode failure (a single point of failure in HDFS v1, solved in HDFS v2 with a Standby NameNode in a High Availability — HA — configuration); and silent data corruption from a stray bit flip (checksums detect it; a healthy replica is served).

A modern alternative to 3× replication is erasure coding (EC), introduced in Apache Hadoop 3.x. EC splits data into data and parity blocks using mathematical algorithms (typically Reed-Solomon coding) and can survive the same number of failures as triple replication while consuming only around 50% storage overhead instead of 200%. The trade-off is more CPU work during reads and writes and slower recovery times.

autofs — on-demand mounting

autofs is the Linux service that mounts file systems only when a user actually tries to access them, and unmounts them automatically after a period of inactivity (five minutes by default). On a university lab of 50 workstations, without autofs an administrator would need to maintain 50 × N /etc/fstab entries for every possible user; with autofs the workstation simply mounts whichever home directory is needed at the moment it is needed.

The service has two parts: the automount daemon (a background process that watches designated directories and triggers mounts as needed), and map files (which tell autofs what to mount and where). The master map at /etc/auto.master lists the top-level directories autofs manages, with their per-directory map files; the per-directory map file (for example, /etc/auto.home) lists individual entries, optionally with a wildcard. The classic wildcard line * -rw,hard,intr nfs-server:/exports/home/& means "for any username typed under /home, mount the matching share from the NFS server, where & is replaced by whatever the user typed." So cd /home/diana causes autofs to mount nfs-server:/exports/home/diana at /home/diana and unmount it again five minutes after Diana stops using it.

There are two kinds of maps. Indirect maps (by far the most common) manage a parent directory and create subdirectories on demand — the pattern shown above. Direct maps specify an absolute mount point for each entry; the master map uses /- as the mount point to signal that this is a direct map. The key commands you should be able to type without thinking are sudo apt install autofs (on Debian or Ubuntu), sudo dnf install autofs (on RHEL, CentOS, or Fedora), and sudo systemctl start autofs && sudo systemctl enable autofs to start the service and have it survive reboots.

Chapter 3 — Data Management and Consistency

Chapter 3

The CAP theorem, in plain words

You get to pick two out of three: Consistency, Availability, Partition tolerance. In the real world the network will partition, so the genuine choice is between CP (Consistency + Partition tolerance) and AP (Availability + Partition tolerance). A pure CA system can only exist on a single machine — and the moment you have a single machine, you no longer have a distributed system to begin with.

In 2000 Eric Brewer drew the three-cornered triangle on a whiteboard at the Principles of Distributed Computing (PODC) conference, hosted by the Association for Computing Machinery (ACM). Two years later Seth Gilbert and Nancy Lynch at the Massachusetts Institute of Technology (MIT) proved the theorem mathematically.

Where the industry actually lands is on the table you should be able to reproduce from memory. MongoDB in its default configuration is CP — it refuses writes during a partition rather than accept conflicting data. Apache Cassandra is AP — it always accepts writes and reconciles later, which is exactly what Facebook needed when they built it for the inbox. etcd, the storage backend of Kubernetes, is CP — the cluster state must be consistent or Kubernetes gets confused. Amazon DynamoDB with eventually consistent reads is AP — a shopping cart can tolerate being a few seconds stale, but a 500-error cannot be tolerated. PostgreSQL with synchronous replication is CP — the leader waits for the replica to acknowledge before replying to the client.

The lesson is that there is no universally correct trade-off. A hospital's patient-record system is very different from a social-media "like" counter; both are correct designs for their respective domains.

Database replication — three families

If CAP is the philosophy, replication is the engineering that makes you pay for your CAP choice. Three families exist, and each maps cleanly to a style of human teamwork.

Single-leader replication (also called master–slave or primary–replica) elects one node as the boss; all writes go to the boss, and the boss streams those writes to the followers. Followers serve reads only. Think of a newspaper: one editor decides the final text, the printing presses copy it. This is the model used by PostgreSQL streaming replication, MySQL async replication, and the default mode of MongoDB Replica Sets.

Multi-leader replication (also called master–master or active–active) lets any node accept writes; each node then gossips its changes to the others. Think of a shared Google Doc with two people typing at once, where the document reconciles in real time. This is the model in CouchDB and Microsoft's Active Directory. The price of multi-leader is conflict resolution — if two clients write to the same record on two different leaders at the same moment, who wins? Timestamps are unreliable across machines; vector clocks are hard; this is the corner of the room where PhD theses get written.

Leaderless replication (also called Dynamo-style after the 2007 Amazon Dynamo paper) treats every node as equal. Writes go to N nodes, reads come from N nodes, and if you arrange W + R > N (write quorum plus read quorum greater than total nodes) you get strong-ish consistency without a boss. This is the model in Cassandra and Riak.

Cross-cutting all three families is the synchronous / asynchronous dial. Synchronous replication has the leader wait for the follower to acknowledge before replying to the client — zero data loss on failover, but the write is only as fast as the slowest follower. Asynchronous replication has the leader reply to the client immediately and stream to the follower in the background — fast, but if the leader dies in the gap you lose the last few writes (this is called replication lag). Most banks run synchronous on the master ledger and asynchronous on the reporting replicas; most social networks run asynchronous everywhere because a 300-millisecond delay on a cat photo is not a crisis.

Eventual consistency — the most misunderstood phrase in the field

If you stop writing and wait long enough, every replica in the system will converge on the same value. It just will not be "right now". Werner Vogels, the Chief Technology Officer (CTO) of Amazon, popularised the term in a 2008 article. His insight: for an enormous class of real problems — likes on a post, views on a video, items in a "recently viewed" list — you genuinely do not need "right now"; you need "pretty soon, and definitely before the user notices".

Three flavours of eventual consistency show up repeatedly: read-your-writes consistency (after you tweet, you see your own tweet immediately, even if everyone else sees it 200 ms later — almost every social network implements this for the logged-in user); monotonic reads (once you see a value, a later read will never show you an older value — it can show you newer or the same, but it cannot go backwards); and session consistency (within one session you get read-your-writes; across sessions, plain eventual).

Classic real-world examples are everywhere: DNS (Domain Name System) propagation takes 24 to 48 hours to be seen globally because every resolver on the planet caches; Instagram's like counter shows the count from the nearest edge cache and converges globally in seconds; the SMTP (Simple Mail Transfer Protocol) email standard is designed around eventual delivery; Git treats your laptop and the central repository as a leaderless replicated system where git pull is the "repair" operation.

PostgreSQL backup — pg_dump and pg_basebackup

Replication protects you from hardware failure. Backup protects you from the intern typing DROP TABLE customers; on the production console at nine in the morning. You need both.

PostgreSQL (PG) ships with two backup tools that solve different problems. pg_dump is the logical backup — it connects to a running server, asks for the data, and writes out a SQL (Structured Query Language) script or a compressed custom-format file. You can pg_restore the dump onto any PostgreSQL server of a compatible version, onto a different operating system, even onto a laptop. Use it for migrating between major PG versions, cloning production into a developer laptop, backing up a single schema or table, or backing up small-to-medium databases (under about 500 GB). Its limits: a huge dump can take hours, and you cannot do Point-In-Time Recovery (PITR) from a dump alone — it is a snapshot, not a continuous archive.

pg_basebackup is the physical backup — it copies the actual files of the PG data directory, page by page, while the server keeps running. The output is a byte-for-byte mirror of $PGDATA. You can only restore it onto the same PG major version and (typically) the same operating system / architecture. Use it for large databases (hundreds of gigabytes and up), as the base of a PITR strategy (combined with Write-Ahead Log — WAL — archiving), and as the initial state of a new streaming replica.

The 3-2-1 rule is the backup commandment every shop follows in some form: three copies of the data, on two different media, with one copy off-site. This rule was born in photography and adopted by the United States Cybersecurity and Infrastructure Security Agency (CISA, formerly US-CERT) because it just works.

Recovery procedures — and the most important habit in this whole course

A backup you have never restored is not a backup. It is a prayer.

PostgreSQL gives you a staircase of recovery options, each more precise than the last. A full restore from pg_dump is good for "yesterday was fine, today is broken". A full restore from pg_basebackup is good for "the whole server is gone". Point-In-Time Recovery (PITR) is the gold standard: you restore the base backup and then replay the Write-Ahead Log (WAL) files up to a specific transaction, a specific timestamp, or a specific named restore point. With PITR you can rewind the database to one second before the intern's DROP TABLE. The magic ingredient is WAL archiving — a setting in postgresql.conf that tells PG to copy every transaction-log segment to a safe location the instant it rotates. Without WAL archiving you can only restore to the moment of the base backup; with it, you can restore to any second in between.

The single practice that separates amateurs from professionals is the restore drill. Once a quarter, pick a random recent backup, restore it onto a spare server, connect, run a few queries, verify row counts against production, and document the elapsed time. If anything is broken, you found it before you needed it, not after.

Chapter 4 — Observability

Chapter 4

The three pillars — logs, metrics, traces

A modern distributed system can put a single user's click through ten or eleven independent services across multiple data centres before that user sees a response. "Go look at the server" stops being a sentence that makes sense, because there is no longer "the server". Observability — the property of a system that lets you ask questions you did not think of in advance — rests on three kinds of data, called the three pillars.

A log is a timestamped, human-readable record of a discrete event ("User 1247 logged in at 14:32:01 from IP — Internet Protocol — 81.92.140.7"). Logs tell you what happened, when, and with what context. Logs come in two flavours: unstructured (just text, easy for humans, hard for machines) and structured (typically JSON — JavaScript Object Notation — slightly noisier for humans, perfect for machines). At any meaningful scale, structured logs win, because the only way to make sense of a million log lines is to query them, and queries require structure.

A metric is a number that varies over time ("CPU usage: 47%", "HTTP requests per second: 1,283", "Active database connections: 17"). Metrics are cheap to store, cheap to graph, and cheap to alert on — you can keep years of metric data in the disk space a single hour of logs would consume, because a metric is just a float and a timestamp. There are four core metric types: counter (only goes up, like an odometer); gauge (goes up and down, like a thermometer); histogram (distributions — "how many requests finished under 100 ms, under 500 ms, under 1 s?", which is what gives you the P99 — 99th-percentile — latency); and summary (similar to a histogram but with percentiles computed on the client side).

A trace is the story of a single request as it travels through your distributed system. Each step is a span — one unit of work with a start time, a duration, and metadata. A trace is a tree of spans. Production systems usually sample traces because storing every trace at full volume is expensive — they typically record one in a hundred, plus 100% of any request that errored. The vendor-neutral standard for emitting traces (and metrics and logs) from your code is OpenTelemetry.

The mental model that ties them together is the slogan you should leave the course with: metrics tell you that something is wrong, traces tell you where it is wrong, and logs tell you why it is wrong. A team with all three has a flashlight in a dark cave; a team with only one is debugging by braille.

Monitoring versus observability

The vocabulary fight you will see online is settled like this: monitoring is asking known questions ("is CPU above 80%? is disk full? is the website returning HTTP — HyperText Transfer Protocol — 200?"). You decide the questions in advance and you watch for the answers. Observability is the property of the system that lets you ask questions you did not think of in advance. You monitor a system; the system is observable (or it is not). The three pillars are how you make a system observable.

Syslog and journald — where logs actually live on Linux

There are two logging systems you absolutely have to know on a Linux box. syslog is the grandfather — born in the 1980s by Eric Allman (the same person who wrote the Sendmail server), formalised by the Internet Engineering Task Force (IETF) in RFC (Request For Comments) 5424 in 2009. The model is simple: a special socket on the system (historically /dev/log) receives messages; a daemon — syslogd, or on modern Linux distributions rsyslog or syslog-ng — reads them and decides what to do (write to a file, forward over the network, or drop).

A syslog message has two tags. The facility says what kind of program emitted it (kern, auth, cron, mail, daemon, local0–local7 for application use). The severity says how bad it is, in order of decreasing seriousness: emerg, alert, crit, err, warning, notice, info, debug. Combined, you get rules that read like English: "send everything from facility mail at severity warning or worse to /var/log/mail-warnings."

Syslog can ship messages over the network — historically over UDP (User Datagram Protocol) on port 514, modernly over TCP (Transmission Control Protocol) on port 514, or with rsyslog over the Reliable Event Logging Protocol (RELP). This is how organisations build centralised log servers; every host on the network ships its logs to one big collector, and Security Operations Centre (SOC) analysts query that one place. Security Information and Event Management (SIEM) products like Splunk, Elastic Security, and Wazuh all speak syslog natively.

systemd-journald is the modern Linux logging daemon, introduced around 2011 with systemd. It stores logs in a binary, indexed format under /var/log/journal/, where every message is a record with structured fields (_SYSTEMD_UNIT, _HOSTNAME, _PID, MESSAGE, plus whatever the program added) and can be cryptographically chained for tamper-evidence. You read it with journalctl. The commands worth memorising are journalctl -xe (show recent log entries with explanatory help — this is the first thing to type when something just broke); journalctl -u nginx.service (show only logs for one systemd unit); journalctl --since "1 hour ago" (time-range filtering); journalctl -p err -b (errors only from the current boot); journalctl -f (follow new entries live); journalctl -o json-pretty -u myapp.service (output as machine-readable JSON).

On most modern Linux systems, journald and rsyslog coexist. journald collects everything first; rsyslog reads from journald via its imjournal input module and forwards messages to the traditional /var/log/* text files and, often, to a remote SIEM. Your logs end up in both places — and that is intentional: journald is excellent locally, but for shipping to remote systems, rsyslog with TCP/RELP is still the workhorse.

Prometheus — the standard, and why pull beats push

By 2012 the industry had a metrics problem. The existing tools — Nagios, Munin, Cacti, Zabbix — were built for long-lived servers with fixed IP addresses; you configured each host into your monitoring tool by hand. Then containers and Kubernetes happened. Suddenly your "hosts" lived for ninety seconds, had random names, and disappeared constantly. The old monitoring world could not keep up.

The same year, engineers at SoundCloud built Prometheus, inspired by Google's internal monitoring system Borgmon. They open-sourced it in 2015. By 2018 it was the second project ever accepted into the Cloud Native Computing Foundation (CNCF) — the first being Kubernetes itself. Today, if you join any engineering team that runs anything in a container, the odds you will encounter Prometheus are very high.

The single most important architectural decision in Prometheus is pull, not push. Most older monitoring tools used the push model — the monitored host ran an agent that pushed metrics to the central server. Prometheus turned this upside down: each application exposes a small HTTP endpoint, typically /metrics, on a known port, and the Prometheus server periodically goes out and scrapes that endpoint, usually every fifteen seconds. The application is passive; the Prometheus server is active. The consequences are large: you can curl /metrics yourself from any laptop to see exactly what Prometheus would see; there is no agent to install on the target; service discovery becomes the central problem (Kubernetes already knows every pod's IP and port, so Prometheus just asks it); failure detection is built in (a silent target stands out immediately, instead of being indistinguishable from a healthy quiet one); and the architecture scales horizontally by giving each Prometheus server a subset of targets.

There is one downside: short-lived batch jobs that finish before Prometheus has scraped them are missed entirely. For those, Prometheus offers a Pushgateway as an intermediary that the job pushes to and Prometheus then scrapes.

A working Prometheus setup has six components you should be able to name. The Prometheus server is the brain — it scrapes targets, stores time-series data, and evaluates rules. Targets are the things being scraped — either applications instrumented with a Prometheus client library or exporters. Exporters are small adapters that translate metrics from systems that do not natively speak Prometheus — the classic example is node_exporter, which exposes Linux machine metrics (CPU, memory, disk, network) on http://host:9100/metrics. The Pushgateway handles short-lived jobs. The Time-Series Database (TSDB) is Prometheus's on-disk storage, highly compressed for the specific shape of monitoring data. PromQL (Prometheus Query Language) is the language you write to query the TSDB.

A typical Prometheus metric line looks like http_requests_total{method="GET",status="200",route="/index"} 1422. Each metric has a name; labels in curly braces carve it into fine-grained series; and the format is dead simple — one per line. A PromQL query reading this might be rate(http_requests_total{status="500"}[5m]), which means "the per-second rate of HTTP 500 responses, averaged over the last five minutes". That kind of expressivity is exactly what made Prometheus famous.

Grafana — turning raw numbers into a picture

Grafana is a web application whose only job is to take data from one or more data sources and turn it into dashboards. It was originally forked from Kibana in 2014 and has grown into the de-facto open-source visualisation tool for engineering metrics. The company that maintains it, Grafana Labs, has been hugely influential in observability over the last decade.

The Grafana vocabulary every student should recognise includes the data source (where Grafana reads from — Prometheus is the most common, but it speaks dozens including InfluxDB, MySQL, PostgreSQL, Elasticsearch, Graphite, Loki for logs, Tempo for traces, even Comma-Separated Values — CSV — files); the dashboard (a page, a set of panels arranged in a grid); the panel (a single visualisation — time series, gauge, stat, table, bar chart, heatmap, geomap); the query (each panel runs one or more queries; for Prometheus, that means PromQL); the variable (a drop-down at the top of a dashboard, such as "environment: prod / staging / dev"); the time range (top-right of the dashboard, global to all panels); the annotation (a vertical line on a chart marking an event such as "we deployed v2.4.1"); and Grafana's own alerting engine, which is a peer to Alertmanager.

There is a short philosophy on good dashboards every junior engineer should hear. Hierarchy — one dashboard for the whole company at the top ("are we alive?"), one per service in the middle, one per concern at the bottom. The four golden signals — Google's Site Reliability Engineering (SRE) book recommends every service have latency, traffic, errors, and saturation on its dashboard; if you can read those four numbers in five seconds, the dashboard is doing its job. No information without action — every panel should answer a question; if you cannot say what action a viewer would take based on the panel, delete the panel.

Alertmanager — the triage nurse of your system

An alert is a rule that fires when a metric crosses some threshold for some duration ("if the rate of HTTP 500s is greater than 1 per second for more than 5 minutes, fire an alert called HighErrorRate"). The rule itself is easy to write. Getting alerts right is hard, because the hard parts are everything that happens after the rule fires: routing (database alerts to the database team, frontend alerts to the frontend team), grouping (when an entire data centre goes down, the on-call engineer should get one notification, not seven hundred), deduplication (if three Prometheus servers all see the same problem, the engineer should still only be paged once), inhibition (if the whole data centre is on fire, the parent alert suppresses the children), silencing (planned maintenance should mute the relevant alerts), and escalation (if the first on-call does not acknowledge in fifteen minutes, page the second on-call).

Prometheus itself does only the first step — evaluating the rule. Everything else is the job of Alertmanager. The pipeline is: the Prometheus server evaluates rules every evaluation_interval (default 15 seconds); when a rule's condition is true and stays true for the rule's for duration, the rule's state goes from pending to firing; Prometheus pushes the firing alert to Alertmanager over HTTP; Alertmanager applies its routing tree to decide which receiver (Slack channel, PagerDuty service, email address, OpsGenie team, generic webhook) should be notified; before sending, it groups alerts that share labels, deduplicates against existing identical alerts, and applies any active inhibitions and silences; eventually a human acknowledges or the rule stops firing, at which point Alertmanager (if configured) sends a resolved notification.

The alertmanager.yml file has four key sections: global (defaults such as the Simple Mail Transfer Protocol — SMTP — server or the Slack URL), route (the routing tree, a recursive structure of routes and sub-routes — the first matching route wins, unless continue: true is set), receivers (named destinations referenced by the route tree), and inhibit_rules ("if alert A is firing, suppress alert B"). Separately, silences are short-term rules created from the Alertmanager web User Interface (UI) for planned maintenance.

The painfully real advice on alerting you should internalise: page on symptoms, not causes (the user is unhappy, not "CPU at 81%"); every alert needs a runbook (a short page that says "if this fires, here is what to check, in order"); tune for low noise (an alerting system that fires 300 times a week with 280 false alarms will be ignored); test your alerts at least once a quarter; and use the for clause so a one-second blip does not wake a human at three in the morning.

Chapter 5 — CI/CD (Continuous Integration / Continuous Delivery)

Chapter 5

What a pipeline actually is

A CI/CD pipeline is an ordered list of steps that runs automatically every time someone changes the code. The first half, Continuous Integration (CI), takes the new code, merges it with the rest of the codebase, compiles it, runs the tests, and tells you within minutes whether the change is broken. The second half, Continuous Delivery / Continuous Deployment (CD), takes that validated build and pushes it through staging environments, packages it, and either drops it at the door of production (Continuous Delivery, with a human approval step) or walks it all the way in (Continuous Deployment, fully automated to production). Same letters, two different stopping points.

The vocabulary you need: a repository is where the code lives — almost always Git, almost always hosted on GitHub, GitLab, or Bitbucket; a commit is the trigger that wakes the pipeline; a build turns source into a runnable artifact (a Java Archive — JAR — for Java, a Docker image for a web app, a binary for Go); the test stage runs unit, integration, and sometimes end-to-end tests; an artifact is what the build produces, stored in a registry such as Docker Hub or JFrog Artifactory; the deploy stage pushes the artifact to dev, staging, or production; a runner (or agent) is the machine that actually executes the pipeline — a Virtual Machine (VM), a container, or a Kubernetes (K8s) pod.

The most common tools in 2026 are GitHub Actions, GitLab CI/CD, Jenkins, CircleCI, and Tekton.

The phrase "shift left" means catching defects as early in the pipeline as possible. A bug caught by an automated test five minutes after a commit costs nothing; the same bug caught by a customer at three in the morning costs a lot.

GitOps — the cluster pulls instead of the pipeline pushing

GitOps inverts the deploy step. Instead of the pipeline reaching into your production cluster and changing things, the cluster watches a Git repository and changes itself. Two reasons that matters: audit trail (every change to production is a Git commit with an author, a timestamp, and a diff — you never have to ask "who changed the replica count last Thursday?"); and recovery (if the cluster gets corrupted, you wipe it, point a fresh one at the same Git repository, and twenty minutes later you have your old cluster back).

The vocabulary: the source of truth is the Git repository — what is in Git is what should be running; if reality differs, reality is wrong. The reconciliation loop is a controller running inside the cluster that wakes up every few minutes, reads the desired state from Git, reads the actual state of the cluster, and makes them match. Pull-based deployment means the cluster pulls changes from Git, so the CI pipeline never has cluster credentials — a real security upgrade over the push model. Drift detection is what happens when someone manually edits the cluster (a sleepy engineer, an attacker, a buggy operator); GitOps notices and either reverts the change or alerts the team. A manifest is a YAML (YAML Ain't Markup Language) file describing a Kubernetes resource — a Deployment, a Service, a ConfigMap — and it is what you commit to the GitOps repository.

The two most common GitOps tools are Argo CD (born at Intuit, now a CNCF-graduated project) and Flux CD (born at Weaveworks, also CNCF-graduated). Most enterprise teams pick Argo CD because of its dashboard, which shows a tree of every resource it manages with sync status and health, in real time.

A subtle point: GitOps does not replace CI. You still need CI to build images and run tests. What changes is the deploy stage. In a non-GitOps setup, the pipeline runs kubectl apply; in a GitOps setup, the pipeline commits a manifest update — for example, bumping image: bicimad/web:1.4.6 to image: bicimad/web:1.4.7 — and Argo CD picks up that commit and applies it. The change still goes to production, but it goes through the Git front door with a clean audit trail.

OpenShift Pipelines (Tekton) — pipelines that live inside Kubernetes

For most of CI/CD's history, the pipeline engine ran on a separate server that held credentials to talk to the cluster. Jenkins is the classic example — a Java application on a VM, holding kubeconfigs and Secure Shell (SSH) keys, deploying into half a dozen environments. That works, but it has well-known problems: the Jenkins server is a single point of failure; scaling is awkward because everything runs on one machine; credentials are centralised and, once stolen, used everywhere.

Tekton flips the model. The pipeline engine is Kubernetes. Each pipeline step is a Pod. The pipeline definition is itself a Kubernetes resource. There is no separate server. If the cluster scales, the CI scales. The vocabulary: a Task is a single unit of work, defined as a Kubernetes resource (each Task runs as a Pod, with each step in the Task running as a container inside that Pod); a Pipeline is an ordered or parallel graph of Tasks (also a Kubernetes resource, defining the workflow but not running by itself); a PipelineRun is an execution of a Pipeline (this is what actually creates the Pods); a Workspace is shared storage between Tasks (cloning a Git repo in Task 1 and building from it in Task 2 requires a workspace so the files persist between Pods); a Trigger listens for events (a Git push, a webhook, a schedule) and starts PipelineRuns automatically; and the Tekton Hub is a public catalog of reusable Tasks (you almost never write git-clone or buildah — a container image build tool — from scratch).

Three reasons teams pick Tekton over GitHub Actions: vendor neutrality (Tekton runs on any Kubernetes cluster, on any cloud, on-premises, or in an air-gapped data centre, while GitHub Actions only runs on GitHub); scale and cost (large enterprises with hundreds of pipelines per day find that running pipelines as Pods on their existing cluster is cheaper than paying for runner minutes); and integration (Tekton plugs into Argo CD natively, lives next to your apps, and shares the same Role-Based Access Control — RBAC — system as everything else).

OpenShift Pipelines is what you get when Red Hat takes Tekton and wraps it with a polished User Interface (UI) inside the OpenShift web console, plus enterprise support and operator-based installation. If your shop runs OpenShift, you almost certainly use OpenShift Pipelines.

Deployment strategies — rolling, blue-green, and friends

Replacing a running production system with a new version is a delicate operation. Do it badly and every user gets an error page for ninety seconds. Do it well and nobody notices anything except that the new feature has appeared. Roughly half a dozen named strategies exist; two are the workhorses.

A rolling deployment replaces Pods one (or a few) at a time. Pod 1 is taken out of the load balancer, the new image is pulled, the new Pod starts, the readiness probe passes, the Pod is added back to the load balancer, and only then does Pod 2 start its own update. At every moment, you have a mix of versions running, but the cluster as a whole is always serving traffic. No downtime, modest infrastructure cost; the downside is no instant rollback (you have to do another rolling deployment to go back). Kubernetes Deployments use rolling updates by default. Two parameters control the speed: maxSurge (how many extra Pods can be created above the desired count) and maxUnavailable (how many Pods can be missing during the rollout).

A blue-green deployment stands up an entire second production environment running the new version (the green) alongside the current one (the blue), tests it thoroughly, and then flips the load balancer — or the DNS entry, or the Kubernetes Ingress — so that all traffic goes to green. Rollback is instant (flip back to blue); downtime is zero. The cost is that during the cutover you are paying for two full production environments. For a stadium ticketing system on a match day, that cost is negligible compared to the cost of an hour of downtime.

A few more strategies worth recognising. Canary deployment releases the new version to 1% of users first, monitors error rates and latency, and gradually ramps up to 5%, 25%, 100%; used heavily for very large user bases. A/B testing deployment is similar to canary but driven by product hypotheses rather than safety. Shadow deployment sends a copy of production traffic to the new version but discards its responses, allowing performance testing without risk to real users. Recreate deployment kills all old Pods and then starts the new ones — causes downtime; only acceptable for non-critical batch jobs or when versions are incompatible.

A summary of trade-offs you should be able to reproduce: rolling has no downtime, slow rollback, low cost, mixed-version risk; blue-green has no downtime, instant rollback, double cost, all-or-nothing risk; canary has no downtime, fast rollback, medium cost, a limited blast zone; recreate has downtime, slow rollback, low cost, an outage window.

Infrastructure as Code — Ansible and Terraform

Infrastructure as Code (IaC) is the practice of describing your servers, networks, databases, and cloud accounts as text files committed to a repository, instead of clicking around in a web console. The text files are the source of truth; a tool reads them and makes the cloud match. Three reasons IaC exists: reproducibility (the same code that built production in Frankfurt builds an identical staging in Madrid in twenty minutes); review (infrastructure changes go through pull requests just like application code, so a senior engineer can review a proposed firewall rule before it lands); and disaster recovery (if a region disappears, you re-run your IaC against another region and you have your environment back).

The two heavyweights are Terraform by HashiCorp and Ansible by Red Hat. They are complements, not competitors — most serious teams use both.

Terraform is for provisioning — creating the cloud resources themselves: VMs, load balancers, databases, S3 (Simple Storage Service) buckets, IAM (Identity and Access Management) roles. Terraform is declarative: you describe the desired end state and Terraform figures out what to create, modify, or destroy to reach it. It uses a language called HCL (HashiCorp Configuration Language), keeps a state file that tracks what it has created, and produces a plan before applying changes — a preview of exactly what it will do.

Ansible is for configuration management — installing and configuring software inside machines that already exist. Install Nginx, configure the database, set up users, deploy an application. Ansible is mostly procedural: you describe a sequence of tasks. It uses YAML files called playbooks, runs over SSH or WinRM (Windows Remote Management), and is agentless — no software needs to be pre-installed on the target machine.

The pattern most teams follow: Terraform creates the cloud infrastructure, then hands the freshly-provisioned VM addresses to Ansible, which configures the operating system and installs the applications.

Two more terms to know: idempotent describes a task that produces the same result whether you run it once or five hundred times (the secret sauce of safe automation — "make sure Nginx is installed" is idempotent; "run apt-get install nginx" can fail the second time); module (Terraform) and role (Ansible) are reusable bundles you write once and use in five environments.

A 2026 licensing note: Terraform's license changed in 2023 to the Business Source License, which prompted the Linux Foundation to fork it into OpenTofu. For most use cases, OpenTofu and Terraform are drop-in compatible.

Case Study 1 — Reorganising a chaotic deployment (Chapter 1)

Case 1

Scenario. A mid-sized fintech in Madrid runs forty microservices on a Kubernetes cluster they built themselves three years ago. Every microservice has its own deployment YAML files, scattered across half a dozen Git repositories. Different teams use different patterns for exposing services externally — some use LoadBalancer Services, some use NodePort, two services use the legacy Ingress controller. New developers take two weeks just to understand which YAML file deploys what. Management is considering moving to OpenShift to bring some order to the chaos. As the platform architect, you are asked to recommend a plan.

Recommended answer. The move to OpenShift is defensible, but it has to be sold honestly. OpenShift is not a magic cure; it is an opinionated distribution of Kubernetes that ships with sane defaults and a polished web console. The actual reasons it would help here are concrete: Routes replace the patchwork of LoadBalancer and Ingress resources with a single, hostname-based external exposure model that every team would learn once; the integrated OpenShift Pipelines (Tekton-based) standardises CI/CD instead of leaving each team to invent its own; the Developer perspective in the web console gives new joiners a visual topology of every microservice in one place; Security Context Constraints (SCCs) force teams to declare what privileges their containers need, which surfaces bad habits early; and OperatorHub integration means teams stop hand-rolling deployment patterns for databases and message queues that an operator could manage. The plan I would propose is a four-phase migration. Phase one: pick one non-critical service, deploy it to an OpenShift cluster running in parallel, document the differences, and let one team learn oc (the OpenShift Command-Line Interface). Phase two: standardise on Routes for every new service, and run a workshop on SCCs so the security team is comfortable. Phase three: migrate services in waves, starting with the noisiest neighbours that currently fight over LoadBalancer IPs. Phase four: turn off the old cluster only when zero traffic is hitting it. The non-negotiable: invest in training before the migration; OpenShift's strict defaults will reject Docker Hub images that worked on the old cluster, and teams need to know why before they hit it at three in the morning.

Case Study 2 — Choosing storage for a media archive (Chapter 2)

Case 2

Scenario. A research project at a Madrid-based archaeology lab needs to store roughly 200 Terabytes (TB) of high-resolution scan data, growing by about 30 TB per year. The data is written once and read rarely (perhaps a few times per year for analysis). The lab has a fixed annual budget. Three options are on the table: a SAN (Storage Area Network) appliance, a NAS (Network Attached Storage) box, and cloud object storage (AWS S3 — Simple Storage Service — or equivalent). The lab director wants your recommendation.

Recommended answer. A SAN is overkill for write-once-read-rarely data — SANs are designed for high-IOPS (Input/Output Operations Per Second), low-latency workloads like databases and virtualisation, and the access patterns here are nothing like that. A NAS box is closer to a fit, but a single NAS appliance for 200 TB plus growth is a capital-expense commitment whose hardware will be obsolete or out of warranty in five years, and a second NAS will be needed for the off-site copy required by the 3-2-1 backup rule. The honest recommendation here is object storage in the cloud, ideally on an "infrequent access" or "archive" storage class such as Amazon S3 Glacier Deep Archive or its equivalent on Azure or GCP. The cost per gigabyte for archive-tier object storage is now in the range of dollars per terabyte per month — far cheaper than buying and maintaining on-premises NAS for cold data. The trade-offs to disclose: read latency on the archive tier is minutes to hours, not milliseconds, which is fine for an annual analysis cycle but unacceptable if the access pattern changes; egress fees apply when you pull the data out, so the budget should explicitly allocate for occasional retrievals; and the lab will need to write a small ingestion script (or use the cloud provider's data-transfer tool — for example, AWS Snowball — for the initial bulk upload). Round it out with the 3-2-1 rule: copy 1 lives in the cloud archive tier; copy 2 is a LTO (Linear Tape-Open) tape library at the lab for catastrophic-cloud-incident recovery; copy 3 is a periodic verification of the cloud copy against the tape checksums. This gives the lab a defensible, predictable cost model and survives the kind of failures that academic institutions actually experience.

Case Study 3 — A banking system that cannot tolerate inconsistency (Chapter 3)

Case 3

Scenario. A regional bank is designing a new core ledger system. Each branch is geographically separate, with intermittent network connectivity between them, especially across the autonomous community boundary. The CIO asks you whether to make the ledger CP, AP, or "some hybrid". The bank is regulated and any double-spend would result in a substantial fine. The CIO also pushes back: "But surely we cannot just refuse customer transactions when the network has a glitch?"

Recommended answer. The ledger has to be CP (Consistency plus Partition tolerance), with no apology. A bank that allows two branches to debit the same account during a partition will eventually pay a regulator more than the entire system would have cost in lost availability over a decade. Refusing to write during a partition is not a failure of the system; it is the system doing its job. The technical implementation should use single-leader synchronous replication for the ledger, on a database that defaults to CP under partition — PostgreSQL with synchronous replication, or MongoDB in its default configuration with majority write concern, would both be defensible. The CIO's concern is fair and is answered by the non-ledger parts of the system: customer-facing read-only services (account balance lookups, transaction history) can run on AP replicas — slightly stale data is acceptable for a balance display, especially with a clear "as of XX:XX UTC" timestamp on the screen; marketing emails, fraud-monitoring alerts, and analytics pipelines can run on eventually consistent secondaries. The pattern is "CP where money moves; AP where people read about money". Round the design out by describing the failure mode plainly to the regulator: during a regional partition, the affected branches cannot write to the ledger but can still answer balance queries from a local read replica with a clear staleness warning; transactions queued during the partition are reconciled and applied when connectivity returns. This is not exotic — it is exactly how etcd protects the state of every Kubernetes cluster in production, and it is exactly the design MIT's Gilbert and Lynch had in mind when they proved the original theorem.

Case Study 4 — Replication strategy for a social application (Chapter 3)

Case 4

Scenario. A social photo-sharing app has six million users worldwide. Users post photos, like each other's photos, and follow each other. The team wants user posts to feel instant (the user must see their own post immediately) and likes to update across the platform "within a few seconds". The data architect on the team is arguing for multi-leader replication so that any region can accept writes locally. You disagree. Defend your position.

Recommended answer. Multi-leader replication is the right tool for a small subset of workloads and the wrong tool for nearly everything else. The right answer here is single-leader asynchronous replication per shard, combined with read-your-writes consistency for the originating user, and eventually consistent propagation of likes and follower counts to other regions. The reasons multi-leader is the wrong choice: it introduces write conflicts on shared records (two users in two regions can race on the same photo's like-count); resolving those conflicts requires either Last-Write-Wins (which silently discards one user's like — bad) or Conflict-free Replicated Data Types (CRDTs, which the team would have to engineer correctly, and which are a research-grade undertaking for any non-trivial data shape — also bad); and the operational complexity of multi-leader systems is well-documented in the original Amazon Dynamo paper and every postmortem since. The right design: each user belongs to a home shard in their nearest region, where the single leader for that shard lives; their writes go to that leader synchronously; their own reads are served from the leader (giving the "I see my own post immediately" guarantee — this is read-your-writes consistency); other users' reads come from local async replicas (giving low latency and eventual visibility of likes). Total propagation budget across the planet: a few seconds, which matches the user experience the product team asked for, and is far below the threshold where users would notice. This design ships every day in production for systems with hundreds of millions of users; reaching for multi-leader is a solution looking for a problem the team does not yet have.

Case Study 5 — Rolling out observability to a struggling team (Chapter 4)

Case 5

Scenario. A platform team has thirty microservices in production. They have no metrics, no centralised logging, and definitely no tracing. Their on-call engineers SSH (Secure Shell) into servers, run grep on log files, and guess. The CTO has given the team a quarter to "fix observability" but has not specified what that means. Where do you start?

Recommended answer. The order matters, because the three pillars are not equally cheap or equally useful at the start. Start with metrics. Stand up Prometheus and Grafana, deploy node_exporter on every host, and instrument every microservice with a Prometheus client library to expose at least the four golden signals (latency, traffic, errors, and saturation, as recommended by Google's Site Reliability Engineering — SRE — book) on a /metrics endpoint. Build one Grafana dashboard per service showing those four numbers and one company-wide dashboard at the top showing "are we alive?". This is the cheapest, highest-value pillar — metrics are tiny on disk, fast to query, and they answer the question "is something wrong?". Next, structured logging. Migrate the team off tail -f and grep by adopting systemd-journald locally and shipping to a central Elastic Stack or Loki cluster. Mandate structured (JSON — JavaScript Object Notation) logs for every new service; agree on a minimum set of fields (ts, level, service, trace_id, msg). Once metrics and logs are flowing, add Alertmanager with a small number of alerts that fire on the four golden signals — and crucially, page on symptoms not causes: a 1-second login page is a symptom; CPU at 81% is a cause and may not even be a problem. Every alert has a runbook. Only then, introduce traces. Instrument the most critical user-journey path (for example, "checkout") with OpenTelemetry, and ship traces to a backend like Jaeger or Tempo. Sample one request in a hundred plus 100% of errors. The result by the end of the quarter: a team that knows when something is wrong (metrics), where it is wrong (traces), and why it is wrong (logs). The slogan is the deliverable: metrics tell you that, traces tell you where, logs tell you why.

Case Study 6 — Replacing a legacy Jenkins setup (Chapter 5)

Case 6

Scenario. A company has been running a single Jenkins server for six years. It has 312 jobs, dozens of plugins (several of which have not been updated since 2019), and one engineer who knows how the whole thing fits together. That engineer is leaving in three months. The CTO asks you to design a replacement. The company already runs OpenShift in production.

Case Study 7 — The night the intern dropped the table (Chapter 3)

Case 7

Scenario. It is 09:02 on a Wednesday morning. An intern has just run DROP TABLE customers; on the production PostgreSQL database. The senior Database Administrator (DBA) is on holiday. You are the most senior person in the room. The company processes about 4,000 customer registrations per day. Walk me through the next sixty minutes.

Recommended answer. First, breathe. Then act in the following order. Minute 0 to 2: stop the bleeding. Take the application's write traffic offline — point the load balancer at a maintenance page, or scale the writer-facing API to zero replicas. This prevents new writes that would conflict with the eventual recovery. Announce in the company chat channel that an incident is in progress and that a senior incident commander has been assigned (yourself, in this case). Minute 2 to 5: assess the damage. Connect to a read replica and verify the scope of the drop. Was it only the customers table or did the intern run something larger? Read the recent pg_stat_activity or audit logs to confirm. Minute 5 to 10: pick the recovery strategy. If WAL (Write-Ahead Log) archiving is enabled and a recent pg_basebackup exists, use PITR (Point-In-Time Recovery): stop PG, rename (do not delete) the broken $PGDATA directory, restore the most recent base backup into a fresh $PGDATA, write a recovery.signal file and a restore_command plus recovery_target_time of one second before the DROP in postgresql.auto.conf, and start PG. PG will replay the WAL files up to that exact second and stop. The customers table is back, missing only the registrations that happened in the gap between the disaster time and the next-completed transaction. If WAL archiving is not enabled, fall back to the most recent pg_dump and accept a larger data-loss window (typically up to one day depending on the backup cadence). Minute 10 to 50: execute the recovery. Time the operation. Document every command you run. Keep the chat channel updated every five minutes with a one-line status. Minute 50 to 60: verify. Connect, run a few sanity queries (SELECT count(*) FROM customers;), compare row counts to known-good metrics from yesterday's analytics, confirm that user-visible flows work. Only then turn writes back on. Post-incident, separately. Write the postmortem within 48 hours. Two action items will be obvious: enable WAL archiving if it was not, and audit how an intern had write privileges on the production database in the first place. The senior DBA returning from holiday should be greeted with a calm, well-documented incident report and a clear set of follow-up actions, not chaos.

Case Study 8 — Adopting GitOps in a regulated environment (Chapter 5)

Case 8

Scenario. A financial services company is subject to strict change-management regulations. Every change to a production system must have an auditable approval trail, a named approver, and a rollback plan. Today, deployments are done by senior engineers running kubectl apply from their laptops, with the approval tracked separately in a ticketing system. The regulator has flagged this as an audit risk. Recommend an architecture that solves the audit problem without slowing the team to a crawl.

Recommended answer. This is what GitOps was invented for, and the regulator has already half-described the solution. The architecture: every change to production is a Git commit to a manifest repository, made via a pull request that requires at least two approvers (the developer and a named senior engineer), reviewed against a written change-control checklist. The pull request body contains the rollback plan as plain text. The Git repository is the source of truth. Argo CD running inside the cluster reads the repository on a reconciliation loop and applies any merged changes to production. No engineer ever has direct cluster credentials; the only path to production is through Git, the pull request, the review, and the merge. What the regulator gets: every change to production has a Git commit hash, an author, a timestamp, a diff showing exactly what changed, a pull request showing who approved it, and a linked ticket. The audit trail is built into the development workflow rather than being maintained in a separate system that gets out of sync. What the team gets: deployments are still fast (Argo CD picks up a merged change within sixty seconds), drift detection is automatic (if anyone manually edits the cluster, Argo CD either reverts it or alerts the team — depending on the selfHeal setting), and the operational complexity of credentials management drops dramatically because no engineer needs cluster access. What you have to invest in: training senior engineers to review manifest changes properly (not rubber-stamp them), wiring Argo CD's notifications into the regulator-approved ticketing system so every sync is logged, and a clear policy for emergency hotfixes (typically: hotfix branch, expedited two-person review, same path through Git). The end state is a workflow the regulator will prefer to the old one, because it produces a stronger audit trail with less paperwork.

Case Study 9 — Replacing a single point of failure in an HDFS cluster (Chapter 2)

Case 9

Scenario. A data-engineering team is running HDFS (Hadoop Distributed File System) v1 with a single NameNode. The NameNode died last month at 02:00, and the cluster was offline for four hours while the team restored from the FsImage and EditLog. The CTO has decreed "this must not happen again." The team has 600 TB of data and 80 DataNodes. Recommend a fix.

Recommended answer. The team needs to upgrade to HDFS v2 (or later) and configure High Availability (HA) for the NameNode. In HDFS HA, there are two NameNode instances at any time: one Active and one Standby. They share their EditLog through a shared storage layer — most commonly a Journal Node quorum of three or five nodes, where every EditLog write must be acknowledged by a majority of journal nodes before the Active NameNode considers it committed. The Standby NameNode reads the EditLog from the journal in real time and keeps its in-memory namespace up to date. When the Active dies, an automatic failover (orchestrated by ZooKeeper with the ZooKeeper Failover Controller, ZKFC) promotes the Standby to Active within tens of seconds. The DataNodes send their heartbeats to both NameNodes, so the new Active immediately has a current view of which blocks live where. Implementation order: pick three machines for journal nodes (ideally on different racks for fault tolerance — the Chapter 2 rack-awareness ideas apply here); deploy a ZooKeeper ensemble of three or five nodes; deploy the Standby NameNode on a new machine; configure dfs.nameservices, dfs.ha.namenodes.<nameservice>, and the journal-node URIs in hdfs-site.xml; configure clients with the same nameservice so they automatically retry the other NameNode on failure; run hdfs zkfc -formatZK and start the ZKFC daemons on both NameNodes. Operational practices that have to come with the upgrade: practice failover at least once a quarter (the same restore-drill discipline from Chapter 3 applies to HDFS HA); monitor the EditLog lag between the journal nodes and the Standby — a lagging Standby that cannot keep up is a hidden disaster; and treat the Secondary NameNode (which is not a backup — only a checkpointer — and which the team almost certainly still has running) as an additional safety net for FsImage / EditLog management, not a substitute for HA. Once HA is in place, schedule a future migration toward erasure coding for cold data tiers to recover some of the 200% storage overhead the cluster pays for triple replication.

Case Study 10 — Taming an alert storm (Chapter 4)

Case 10

Scenario. A company's on-call engineers are exhausted. They receive an average of 47 pages per day, of which roughly 40 turn out to be false alarms or duplicates. Two engineers have resigned in three months citing burnout. The Alertmanager configuration has 312 alert rules. The team agrees the system is broken. As the new Site Reliability Engineering (SRE) lead, you have one quarter to fix it.

Recommended answer. Start with the philosophy and then attack the configuration. The philosophy, repeated from Chapter 4: page on symptoms, not causes. A page should mean a user is unhappy right now, not a metric is interesting. "CPU at 81%" is not a page; "login page latency above 1 second for 5 minutes" is. Phase one (week 1 to 3): the inventory. List every one of the 312 alerts. For each, answer two questions: Has this alert fired in the last 90 days? If yes, did it lead to action? Alerts that have not fired in 90 days are deleted. Alerts that fire but never lead to action are deleted or downgraded to a non-paging severity. This usually reveals that 60% to 80% of an over-grown rule set is noise. Phase two (week 4 to 6): the symptoms. For each surviving alert, ask: does this alert correspond to a user-visible symptom? If yes, keep it and rewrite the alert text to describe the symptom (HighCheckoutLatency, not CPULoadOverThreshold). If no, downgrade to a dashboard-only signal. Pages should be for symptoms. Phase three (week 7 to 9): the routing. Configure Alertmanager's route tree to send database alerts to the database team, frontend alerts to the frontend team, and so on, using severity: critical | warning | info labels on every rule. Configure group_by, group_wait, and group_interval so the on-call engineer gets one notification per incident, not seven hundred. Add inhibit_rules so a parent alert (DatacenterDown) suppresses the children. Phase four (week 10 to 12): the runbooks and the testing. Every surviving alert needs a runbook — a short page that says "if this fires, here is what to check, in order, and what to do". Without runbooks, alerts are how on-call engineers quit. Quarterly, the team deliberately triggers every critical alert in a staging environment and confirms the page lands where it should. Phase five: the metric. Track alerts per engineer per day as a team-level Key Performance Indicator (KPI). The goal is fewer than five pages per engineer per week, of which at least four are real. When the team hits that target, you have rebuilt trust in the system, and the engineers stop resigning.

According to the CAP theorem, in the presence of a network partition, which two properties must a distributed system trade off between?

A. Capacity and Availability B. Consistency and Availability C. Cost and Atomicity D. Concurrency and Atomicity

Answer: B — Consistency and Availability

Explanation. The CAP theorem states that a distributed system can simultaneously provide at most two of Consistency, Availability, and Partition tolerance. Because real-world networks will partition, the operationally meaningful choice is between CP (refuse writes during a partition to preserve consistency) and AP (keep accepting writes to preserve availability, and reconcile later). A pure CA system would have no partition tolerance, which in practice means a single machine — and a single machine is not a distributed system.

MCQ 20 — Chapter 3 (Data Management and Consistency)

A bank is designing its core ledger and chooses MongoDB in its default configuration. Under the CAP theorem, which trade-off has it chosen?

A. AP — sacrificing consistency to keep accepting writes during a partition B. CP — sacrificing availability to refuse conflicting writes during a partition C. CA — guaranteeing both consistency and availability with no partitions D. None — MongoDB is a single-machine database

Answer: B — CP

Explanation. MongoDB in its default configuration is CP: during a partition it refuses writes rather than risking inconsistent data. This is the correct trade-off for any system where the cost of inconsistent state (a double-spend on a bank ledger, an over-sold seat on an airline) is higher than the cost of a brief refusal to write. Cassandra would have been the AP alternative — always accepts writes, reconciles later — which is the right choice for systems like a "likes" counter where missing data for a few seconds is harmless.

MCQ 21 — Chapter 3 (Data Management and Consistency)

In single-leader replication with synchronous replication, what is the price the system pays for "zero data loss on failover"?

A. The leader can be queried for reads from any client B. Each write takes as long as the slowest follower's acknowledgement C. The followers eventually accept writes when the leader is unavailable D. The cluster cannot tolerate a network partition

Answer: B

Explanation. Synchronous replication requires the leader to wait for the follower(s) to acknowledge the write before responding to the client. This eliminates replication lag at the moment of failover — if the leader dies, the follower has the same data — but the write is only as fast as the slowest follower. Asynchronous replication lets the leader reply immediately and stream to followers in the background, which is fast but loses any unreplicated writes on failover.

MCQ 22 — Chapter 3 (Data Management and Consistency)

Which form of eventual consistency guarantees that a user, immediately after writing a value, always sees their own most recent write — even if other users see a slightly stale value for a brief moment?

A. Monotonic reads consistency B. Read-your-writes consistency C. Strong consistency D. Causal consistency

Answer: B — Read-your-writes consistency

Explanation. Read-your-writes consistency guarantees that the writer's subsequent reads always reflect their own most recent write. Most social platforms implement this by routing the writer's reads to the leader or to a session-pinned replica. Monotonic reads is the related but different guarantee that a reader never sees a value older than one they have already seen (no time travel backwards). Strong consistency is the all-or-nothing version where every read sees the latest write from any client — much more expensive.

MCQ 25 — Chapter 3 (Data Management and Consistency)

A PostgreSQL administrator wants the ability to restore the database to any specific second in the recent past (for example, one second before an intern dropped a table). Which combination of features must be enabled?

A. pg_dump only B. pg_basebackup only C. A recent pg_basebackup plus continuous WAL (Write-Ahead Log) archiving D. Synchronous replication to a hot standby

Answer: C

Explanation. Point-In-Time Recovery (PITR) requires both a base backup (from pg_basebackup) and a continuous archive of the Write-Ahead Log (WAL) segments. During recovery the administrator restores the base backup, then has PostgreSQL replay the WAL files up to the target time (recovery_target_time in postgresql.auto.conf) at which point the database stops. Essay 02 — Chapter 1 (Container Orchestration with OpenShift)

Question. Explain what etcd is, why it sits at the heart of every Kubernetes and OpenShift cluster, and what operational consequences follow from that role.

Model answer. etcd is a distributed, strongly consistent key-value store that uses the Raft consensus protocol to keep its replicas in agreement. In Kubernetes and OpenShift, every piece of cluster state — every Deployment, Pod, Service, Secret, Role binding — is ultimately a row in etcd. The API server is essentially a typed, authenticated, validating front door to etcd. Three consequences follow. First, etcd availability equals control-plane availability. If a majority of etcd members are unreachable, the API server refuses writes, which means no new Pods get scheduled, no rollouts progress, and oc apply fails — although already-running Pods keep serving traffic. Second, etcd is sensitive to disk latency and network jitter, because Raft requires committing to a majority before a write is acknowledged. Production etcd is run on dedicated low-latency local SSDs, an odd number of members (3, 5, or 7), and ideally on dedicated nodes rather than co-tenant with noisy workloads. Third, etcd must be backed up — losing etcd without a backup means losing the description of the cluster. Standard practice is scheduled etcdctl snapshot save, stored off-cluster, with restore drills proving the snapshot actually works.

Essay 03 — Chapter 1 (Container Orchestration with OpenShift)

Essay 04 — Chapter 1 (Container Orchestration with OpenShift)

Question. A junior engineer asks: "Why don't we just put all the application Pods on the master nodes? We have spare CPU there." Give the principled answer.

Model answer. The master (control-plane) nodes run the components that decide what should happen in the cluster: the API server, the scheduler, the controller-manager, and the etcd data store. If a malfunctioning application Pod consumes all the CPU or memory on a master, the consequences are not "one slow application" — they are "the cluster can no longer schedule, restart, or repair anything." That is a vastly larger blast radius than the same Pod misbehaving on a worker. There are also security considerations: the control plane handles tokens, certificates, and the API server's signing key; mixing in arbitrary application workloads enlarges the attack surface. And there is a reliability argument: control-plane components have specific performance characteristics — etcd is sensitive to disk fsync latency, for example — that compete badly with the bursty resource patterns of typical applications. The principled architecture, codified by default in OpenShift, is to taint master nodes so that ordinary workloads do not land on them, run dedicated infrastructure nodes for things like the registry and monitoring stack, and reserve worker nodes for the application Pods that pay the bills. The "spare CPU on the masters" is a feature, not a waste — it is the headroom the cluster needs to keep itself stable when load spikes.

Essay 06 — Chapter 2 (Distributed File Systems)

Question. Explain the role of the NameNode in HDFS, what would happen if it failed, and how production deployments mitigate that risk.

Model answer. The NameNode is the metadata server of HDFS. It maintains the directory tree of the file system, the permissions on each file and directory, and — crucially — the mapping from each file to its constituent 128 MB blocks and the DataNodes those blocks live on. None of the actual file content lives on the NameNode; it lives on the DataNodes. The trade-off is that every file open, list, or block lookup goes through the NameNode, which makes it both a performance bottleneck for small-file workloads and a single point of failure (SPOF). If the NameNode dies unexpectedly without high availability, two things happen: clients can no longer create, open, or list files (although already-streaming reads of known block locations can sometimes complete), and on restart the NameNode must replay the FsImage plus the EditLog before becoming available again, which on a large cluster can take many minutes. Production deployments mitigate this with HDFS High Availability: two NameNodes (Active and Standby) keep their edit logs synchronised via a quorum of JournalNodes (typically three or five), and ZooKeeper plus the ZooKeeper Failover Controller decide when to promote the Standby to Active. DataNodes heartbeat to both NameNodes so the Standby has a current block map and can take over within seconds. The historical Secondary NameNode is not an HA mechanism; it merely checkpoints the FsImage to bound EditLog growth. Confusing the two is a classic interview-trap mistake.

Essay 07 — Chapter 2 (Distributed File Systems)

Question. Distinguish file storage, block storage, and object storage. Give one realistic example workload for each in a typical mid-sized Madrid company.

Model answer. Block storage presents a raw, fixed-size sequence of blocks (a device) to the operating system, which formats it with a file system. It is the lowest level, has the lowest latency, and is what databases and virtual-machine disks want. A typical Madrid use case: the PostgreSQL cluster backing the company's billing application runs on iSCSI Logical Unit Numbers (LUNs) from a Storage Area Network (SAN), giving the DBAs low-latency block devices that they format with XFS. File storage exposes a directory tree over a network protocol such as the Network File System (NFS) or Server Message Block (SMB). Many clients can mount the same share and see the same files. A typical use case: the engineering team's shared project drive — designs, internal documentation, Computer-Aided Design (CAD) files — lives on a Network-Attached Storage (NAS) device exported over NFS. Object storage exposes a flat namespace of immutable objects through an HTTP REST API such as Amazon S3's. It scales to petabytes and is the cheapest tier per gigabyte. A typical use case: the marketing team's video archive and the long-term retention of compliance logs (e.g., five years of access logs from the customer portal) live in a bucket. The mental model that helps students remember the distinction is: block is what the kernel sees, file is what users see, and object is what HTTP sees.

Essay 08 — Chapter 2 (Distributed File Systems)

Question. Explain how replication, checksums, and erasure coding work together to provide durability in a modern distributed file system.

Model answer. Each mechanism attacks a different failure mode. Replication stores N copies of each block (HDFS defaults to three). If one copy is lost — disk failure, node failure, rack failure — another copy is used while the system creates a fresh replica on a different node. Replication is simple, fast to read from, and expensive in raw capacity: 3× replication uses three times the disk for the same logical data. Checksums stored alongside each block (typically CRC-32C or stronger) let the storage layer detect silent corruption — bit rot, a bad controller, a cosmic-ray flip — that replication alone would not notice, because the corrupt copy would otherwise look fine. When a checksum mismatch is found, the bad copy is discarded and re-replicated from a good one. Erasure coding replaces N-way replication with a mathematical scheme (for example, Reed–Solomon 10+4) that adds parity fragments so that any 10 of the 14 fragments can reconstruct the original data. Erasure coding gives durability comparable to triple replication at roughly half the storage overhead, but reads are slower if a fragment is missing (decoding cost) and writes touch more nodes. In practice, hot frequently read data is kept replicated, while cold rarely read data is erasure-coded — a pattern visible in HDFS Erasure Coding, Ceph pools, and cloud object storage tiers alike. Together, the three techniques cover three threats: hardware failure (replication and erasure coding), silent corruption (checksums), and cost (erasure coding tames the price of large cold archives).

Essay 09 — Chapter 3 (Data Management and Consistency)

Question. State the CAP theorem precisely, explain why the popular "pick two of three" summary is misleading, and use a concrete example to illustrate the real choice.

Model answer. The CAP theorem, proposed by Eric Brewer at the Association for Computing Machinery Principles of Distributed Computing (PODC) symposium in 2000 and proved by Gilbert and Lynch at MIT in 2002, says: in a distributed system that experiences a network Partition, you cannot simultaneously provide Consistency (every read sees the most recent write) and Availability (every request gets a non-error response). The "pick two of three" tagline misleads because partitions are not optional — in a real network, partitions happen. You do not get to "pick CA and abandon P"; the network does not consult you. The honest reading is: when the network is healthy, you serve both C and A; when a partition occurs, you must choose between rejecting requests (preserving C, sacrificing A) and serving possibly stale or conflicting data (preserving A, sacrificing C). A concrete example: imagine a banking ledger replicated between Madrid and Frankfurt. If the link between the two cities fails, a CP system such as etcd or a synchronously replicated PostgreSQL primary refuses writes on the side that cannot reach a majority, preserving correctness at the cost of an outage in that region. An AP system such as Cassandra in its default configuration accepts writes on both sides, returning success immediately, and reconciles when the partition heals — possibly producing two reads that briefly disagree about a balance. Banks pick CP; shopping carts pick AP. The choice is about which lie you are willing to tell the user during a partition: "service unavailable, please retry" or "your data here may be slightly out of date."

Essay 10 — Chapter 3 (Data Management and Consistency)

Question. Compare single-leader, multi-leader, and leaderless replication. For each, name one production-grade system that uses it, one strength, and one significant weakness.

Model answer. Single-leader (primary–replica) replication elects one leader that accepts all writes; replicas pull or stream changes and serve reads. PostgreSQL streaming replication is the canonical example. The strength is conceptual simplicity: there is one source of truth, no write conflicts, and reasoning about consistency is straightforward. The weakness is write scalability and failover complexity: the leader is a single bottleneck for writes, and choosing the next leader after a failure is a hard problem that introduces tools such as Patroni plus a distributed lock store. Multi-leader replication accepts writes on more than one node and replicates between leaders. Active–active geographic deployments and CouchDB replication are examples. The strength is availability and locality: each region writes locally and stays available during inter-region partitions. The weakness is conflict resolution: when two leaders accept conflicting edits to the same row, somebody has to decide which wins (last-write-wins, application-level merge, Conflict-free Replicated Data Types). Leaderless replication, as described in the Amazon Dynamo paper (2007) and implemented in Cassandra and Amazon DynamoDB, accepts writes on any replica and uses read and write quorums (R + W > N) to give probabilistic consistency. The strength is availability and horizontal write scalability. The weakness is operational complexity: tuning quorums, handling hinted handoff and read repair, and explaining vector-clock conflicts to application developers all add cognitive load.

Essay 11 — Chapter 3 (Data Management and Consistency)

Question. Explain what eventual consistency means, why a developer might find it surprising in practice, and which weaker-but-stronger-than-eventual models — described by Werner Vogels in 2008 — make life more bearable.

Model answer. Eventual consistency is the guarantee that, if no further writes occur, all replicas will eventually converge to the same value. The model says nothing about when — convergence could take milliseconds or seconds — and nothing about what happens to readers in between. Developers are surprised because the same user can write a new value, read it back, and see the old one, because their read happened to land on a replica that had not yet caught up. The cure is not full linearizability (often too expensive); it is one of the named intermediate models. Read-your-writes guarantees that after a client writes a value, that same client never reads an older one — implemented by routing the client's reads to the leader for a window, or by passing a version cookie. Monotonic reads guarantees that successive reads by the same client never go backwards in time — implemented by pinning the client to a particular replica or set of replicas. Monotonic writes guarantees that writes by the same client are seen by all replicas in the order the client issued them. Session consistency is read-your-writes plus monotonic reads plus monotonic writes within a single session — the practical sweet spot for most user-facing applications. Picking one of these models lets a system stay AP for availability while still feeling correct from the perspective of any individual user.

Essay 12 — Chapter 3 (Data Management and Consistency)

Question. A small e-commerce company in Madrid runs a PostgreSQL database holding its orders. Outline a backup and recovery strategy that meets the 3-2-1 rule recommended by CISA and supports Point-in-Time Recovery (PITR) to any minute in the last 14 days. Mention the specific tools.

Model answer. PostgreSQL provides exactly the primitives needed. The strategy has three layers. First, a daily physical base backup with pg_basebackup captures a consistent snapshot of the data directory while the database stays online; these snapshots are retained for 14 days and form the foundation of every restore. Second, continuous Write-Ahead Log (WAL) archiving — configured via archive_mode = on and an archive_command that copies each WAL segment off the database server as soon as it is closed — captures every committed change. Together, base backup plus WAL log replay gives PITR: pick any moment in the last 14 days, restore the most recent base backup taken before that moment, then replay WAL segments until the target time, and stop. Third, occasional logical dumps with pg_dump provide a version-portable export, useful for restoring a single schema or moving between major versions. To satisfy the 3-2-1 rule, the backups exist in three copies (base backups + WAL archive on the backup server; a mirrored copy on a second on-premises disk; an encrypted copy synchronised to a cloud bucket such as Amazon S3 Object Lock or Backblaze B2), on at least two media types (local disk plus cloud object storage), and with at least one copy off-site (the cloud bucket, in a different geographic region). Critically, a monthly restore drill verifies that a PITR to an arbitrary minute actually succeeds — an untested backup is a wish, not a backup. The modern extension "3-2-1-1-0" adds one immutable copy (S3 Object Lock or a Write-Once-Read-Many archive) and zero errors on the latest restore test.

Essay 13 — Chapter 4 (Observability)

Question. Define the three pillars of observability and explain how a Site Reliability Engineer (SRE) on a Tuesday afternoon would use each pillar during a real incident.

Model answer. The three pillars are metrics, logs, and traces. The mnemonic is: metrics tell you that something is wrong, traces tell you where, logs tell you why. Suppose at 15:42 on a Tuesday a Prometheus alert fires: the 99th-percentile latency on the checkout service has crossed 800 ms for five minutes. The SRE opens the on-call Grafana dashboard for checkout and uses metrics to see the shape of the problem: error rate is normal, request rate is flat, but latency on the POST /confirm-order endpoint has doubled. That answers "is it actually broken and how badly" — the that. The SRE then opens the distributed tracing tool (e.g., Jaeger or Tempo) and pulls a sample of recent slow traces. The traces show the request fans out across six services, but almost all the added latency comes from a single span: payments → fraud-check. That is the where. Finally, the SRE pulls logs from the fraud-check Pod for the last 10 minutes via journalctl or Loki and discovers the service has been logging connection timeout to fraud-rules-db, retrying in 500ms since 15:36. That is the why — the downstream rules database is unreachable. Notice how each pillar without the others would have left the SRE staring at half a problem: metrics alone identify symptoms but not cause; logs alone are an unsearchable ocean of events; traces alone do not say which logs to read. The pillars are stronger as a stack than as alternatives.

Essay 14 — Chapter 4 (Observability)

Question. Describe the architecture of Prometheus, including its data model, collection model, components, and a typical reason a team might add a separate Pushgateway.

Model answer. Prometheus is a Cloud Native Computing Foundation (CNCF) graduated project, originally open-sourced by SoundCloud in 2012. Its data model is a multi-dimensional time series: each metric has a name (http_requests_total), a set of labels (method="POST", status="500", route="/checkout"), and a stream of timestamped numeric samples. The label dimension is what makes the model expressive — rather than a separate metric per HTTP route, the route is just a label, and PromQL lets you slice by it. The collection model is pull: the Prometheus server has a list of scrape targets and, on a configurable interval (typically 15–30 s), makes HTTP GET requests to each target's /metrics endpoint. Pull has practical benefits — the server controls scrape frequency, easily detects "target down" because the scrape fails, and avoids the need for clients to know about the monitoring system. The components are: the Prometheus server (scrapes, stores, evaluates rules); targets (the applications and exporters that publish metrics — node_exporter for Linux hosts, mysqld_exporter for MySQL, and so on); the local time-series database (TSDB) on the server's disk; PromQL as the query language; Alertmanager as the downstream that handles routing and silencing of alerts; and Grafana (separate project) as the typical visualization layer. The Pushgateway exists to bridge the gap for short-lived batch jobs that finish before they can be scraped — a nightly backup script runs for 30 seconds, pushes its result metrics to the Pushgateway, exits, and Prometheus scrapes the Pushgateway like any other target. It is an intentional exception, not a general substitute for the pull model.

Essay 15 — Chapter 4 (Observability)

Question. Why is the principle "page on symptoms, not on causes" central to good alert design? Illustrate with an example of a bad alert and a better one.

Model answer. Cause-based alerts try to catch every internal condition that might be a problem — the disk on node 7 is at 71% full, the cache hit rate dipped, a thread pool is half-saturated — most of which never affect anyone. They produce alert fatigue: dozens of pages a week, almost all noise, and after a month the on-call engineer stops reading them, which is exactly when the real outage slips through. Symptom-based alerts page only when something users actually notice: latency above an agreed threshold, error rate above a budget, success rate of a critical user journey below target. There are fewer of them, every one is real, and they map directly to the Google SRE four golden signals (latency, traffic, errors, saturation). Bad alert (cause-based): "CPU on the payments Pod is above 80% for 2 minutes." This fires harmlessly during every traffic peak, and not firing during a real outage where the Pod is wedged with low CPU. Better alert (symptom-based): "The 99th-percentile latency of POST /checkout has been above 1000 ms for 5 minutes, and the request rate is non-trivial." This fires only when the customer-facing journey is actually broken, regardless of which underlying resource is the culprit, and the for: 5m clause filters transient blips. The cause-based observation can still exist as a non-paging warning that engineers consult during business hours — useful information, but not a 03:00 wake-up call.

Essay 16 — Chapter 4 (Observability)

Question. A startup just rolled out Grafana and connected it to Prometheus. What is the minimum dashboard they should build before going to bed, and why those panels specifically?

Model answer. The dashboard should answer one question: "is the service serving its users right now?" Anything that does not contribute to that answer is decoration and can wait. Concretely, the dashboard contains the four golden signals from the Google SRE book, one panel each. Latency — a graph of the 50th, 95th, and 99th percentiles of the main user-facing endpoint, because averages hide tail problems; if the 99th percentile is fine, almost everyone is having a fast experience. Traffic — requests per second, segmented by endpoint and status code, so the on-call can tell apart "the world has gone quiet" (broken upstream load balancer) from "the world is on fire" (Black Friday). Errors — the ratio of 5xx (and 4xx, separately) to total requests; this is the most reliable single indicator that something is genuinely broken rather than slow. Saturation — a measure of how full the most constrained resource is, typically CPU utilisation, memory usage, and queue depth on the busiest pool; saturation is the leading indicator that warns before latency and errors get visibly bad. Add to those a clearly visible time range selector (default last 1 hour), a variable for environment (prod / staging), and an annotation layer that overlays recent deployments onto every panel — so the question "did this start when we deployed?" can be answered at a glance. Anything else — JVM heap histograms, per-endpoint heat maps, business KPIs — can grow on top of this once the basics exist.

Essay 17 — Chapter 5 (Continuous Integration / Continuous Delivery)

Question. Define Continuous Integration, Continuous Delivery, and Continuous Deployment. Where exactly does the line between them fall, and what cultural commitments does each step require from the team?

Model answer. Continuous Integration (CI) is the practice of integrating every developer's work into the shared mainline frequently — at least daily, and on every push in a mature setup — with automated builds and tests gating the merge. The cultural commitment is that the build is always green and that breaking it is everyone's stop-the-line problem. Continuous Delivery (CD-delivery) extends CI: every green build produces a deployable artifact, and a path to production exists that requires only a human approval to release. The commitment is that the team keeps the codebase in a release-ready state at all times — no week-long stabilisation branches, no "we'll fix it before the release." Continuous Deployment (CD-deployment) removes the human approval: every green change goes to production automatically, often within minutes of commit. The commitment is significantly heavier: automated tests have to be trusted enough that nobody reviews each release, deployment must be reversible quickly (feature flags, instant rollback, canary releases), and monitoring must catch in production what the tests missed. The line between the two flavours of CD is exactly the manual gate. Most regulated industries — banking, healthcare — stop at Continuous Delivery, because the audit trail of "human approved this release" is part of their compliance posture. Many consumer software companies push through to Continuous Deployment, because the cost of a small bug reaching one percent of users for ten minutes is less than the cost of slower iteration.

Essay 18 — Chapter 5 (CI/CD)

Question. Compare rolling, blue–green, and canary deployment strategies. For each, give one realistic situation in which it is the right choice, and one significant downside.

Model answer. Rolling deployment replaces instances of the old version with the new one a few at a time — controlled in OpenShift by maxSurge and maxUnavailable. Good when: the application is stateless, backward-compatible with the previous version during the rollout window, and the team wants the simplest possible default. Downside: during the rollout, both versions are serving traffic simultaneously, so anything that depends on "all instances are on the new version" — a schema change that drops a column, a behavioural change that depends on no callers still using the old API — will misbehave for the duration of the roll. Blue–green deployment keeps the old version (blue) running while a complete copy of the new version (green) is brought up alongside, then the load balancer or router is flipped from blue to green. Good when: the team wants atomic cutover and instant rollback (flip the router back to blue), and the budget tolerates running double infrastructure during the cutover window. Downside: it requires roughly 2× the infrastructure for the duration, and database schema changes still have to be co-deploy-safe because at the moment of the cutover both versions briefly hit the same data store. Canary deployment routes a small fraction of traffic — 1%, then 5%, then 25%, then 100% — to the new version, watches metrics at each step, and rolls back if symptoms appear. Good when: the team has solid metrics and the change is risky enough that exposing it gradually is worth the complexity (e.g., a new ranking algorithm). Downside: it requires the routing infrastructure to split traffic deterministically and the observability infrastructure to compare canary versus baseline meaningfully; without those, canary is just "a smaller rolling deployment" and provides no real safety benefit.

Essay 19 — Chapter 5 (CI/CD)

Question. Explain the GitOps model. Why is it described as a "pull-based" approach to deployments, and what audit and security benefits does it deliver compared with a traditional kubectl apply from a developer's laptop?

Model answer. GitOps declares the desired state of an environment in a Git repository — manifests for every Deployment, ConfigMap, Secret reference, Service, and Route — and runs an in-cluster controller (such as Argo CD or Flux CD) that continuously reconciles the live cluster against the repository. The controller pulls the desired state from Git on a short interval and compares it to what is actually running; when they differ, it either auto-corrects (selfHeal: true) or surfaces the drift as an "OutOfSync" alarm. It is called pull-based because the controller initiates the read from Git rather than a developer pushing into the cluster from outside — the cluster reaches out to its source of truth, not the other way around. The audit and security benefits are substantial. Audit: every change is a commit, with author, timestamp, and the diff that was applied; pull requests give a peer-review gate and a place for compliance reviewers to leave comments before the change reaches production; rollback is git revert rather than scraping through Slack to remember what was deployed. Security: the cluster's API endpoint does not need to be reachable from developer laptops or the public Internet for deployment purposes — the controller pulls outward to Git, so the attack surface is narrower; long-lived kubeconfig credentials with cluster-admin do not need to be distributed to developers; and the destination namespaces are constrained by the Argo CD or Flux RBAC, not by what each developer happens to have permission to do. The combination of git-as-the-only-deployment-path, peer review on every change, and a narrowed cluster attack surface is why regulated environments find GitOps strictly easier to defend than ad-hoc kubectl apply.

Essay 20 — Chapter 5 (CI/CD)

Question. A team has a Terraform repository that provisions the VMs and networks for their environment, and an Ansible repository that configures the OS and installs the application runtime inside those VMs. Explain why this split is sensible, and what each tool's idempotence model means for re-running the pipeline twice on the same day.

Model answer. The split aligns with how each tool was designed. Terraform is declarative and works against the provider APIs of cloud and on-premises platforms — Amazon Web Services, Azure, Google Cloud, vCenter, OpenStack, and dozens more. Its job is to bring infrastructure-as-a-resource (a virtual machine, a subnet, a security group, a load balancer) into the shape described by the HashiCorp Configuration Language (HCL) code. Ansible is procedural by default — playbooks are sequences of tasks — and works inside the operating system over SSH or Windows Remote Management (WinRM). Its job is to configure that operating system: install packages, edit configuration files, manage services, deploy application binaries. Trying to do "install nginx and copy this config file" with Terraform is awkward (it has provisioners, but they are discouraged), and trying to do "create 12 EC2 instances with the right tags" with Ansible's cloud modules works but loses Terraform's state-aware diffing. So the team uses each for what it is best at: Terraform shapes the platform, Ansible shapes the contents of each VM. Idempotence means that running the same code twice on the same target produces the same end state without unintended side-effects on the second run. For Terraform, this is enforced by the state file: on the second run, Terraform compares desired state (the HCL) to recorded state (the state file) to actual state (queried from the cloud API), and only applies the differences — if nothing changed, the run is a no-op. For Ansible, idempotence is a property each module strives to provide: the apt module installs a package only if it is missing, the template module writes a file only if its content would change, the service module starts a service only if it is not already running. The cultural payoff is that pipelines become safe to run repeatedly — no fear of "what if I trigger this twice by accident" — which in turn is what makes CD pipelines, drift detection, and disaster-recovery rebuilds possible at all.
