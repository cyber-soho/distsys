Observability: Logs, Metrics, and Traces

Key Concepts Review

Click on a concept to reveal its definition.

1. Monitoring
Watching a known set of indicators to answer if something expected to break is broken.
2. Observability
The property of a system that lets you ask new questions about its behavior from the outside without shipping new code.
3. Primary use case for logs
Forensics, audit trails, and answering exactly what a user clicked or did.
4. Primary use case for metrics
Trend graphs, alerting, and high-level aggregations over time.
5. Primary use case for traces
Identifying exactly where in a complex call graph latency or errors are occurring.
6. Correlation ID (trace_id)
A unique identifier attached to a request that binds its logs, metrics, and traces together.
7. syslog
The original Unix logging protocol that stores plain text logs based on facility and severity.
8. journald
A systemd component that stores structured, binary, and indexed log records.
9. Most useful journalctl command for viewing specific unit logs
journalctl -u [unit name]
10. Standard syslog severity threshold for production alerts
Level 3 (err) - Error conditions.
11. Prometheus data collection architecture
Pull-based; it periodically scrapes an HTTP endpoint (/metrics) on every target.
12. Prometheus Exporters
Components that translate non-native data sources into the Prometheus exposition format.
13. Counter metric
A numeric measurement that only goes up, representing cumulative events, usually queried with a rate() function.
14. Gauge metric
A numeric measurement that can go up and down, representing a snapshot of current state like queue depth.
15. Histogram metric
A metric type that groups observations into buckets, primarily used for calculating server-side percentiles like p99 latency.
16. Primary risk of using unbounded values in metric labels
Cardinality explosion, which can crash the Time-Series Database (TSDB).
17. Grafana
A datasource-agnostic visualization layer that owns the display and alerting UI but no underlying storage.
18. Ideal design for the top row of a Grafana dashboard
A big number or traffic-light color indicating if the system is generally okay, readable in 10 seconds.
19. Core function of Alertmanager
To protect humans from alert fatigue through grouping, inhibition, silencing, and routing.
20. Three mandatory properties of a production alert
It must be actionable, symptom-based (describing user-visible pain), and documented with a runbook.
21. RED golden signals
Rate, Errors, Duration (used primarily for monitoring services).
22. USE golden signals
Utilisation, Saturation, Errors (used primarily for monitoring hosts).
23. TSDB
Time-Series Database, the purpose-built storage mechanism used by tools like Prometheus.
24. Alerting inhibition
Suppressing downstream alerts when a broader system failure (like a datacenter outage) is already firing.
25. OpenTelemetry (OTel)
A vendor-neutral instrumentation standard and SDK used primarily to generate and route trace data.

1. An e-commerce system is experiencing slow load times specifically for users in Spain on Tuesdays. Which discipline provides the tools required to investigate this novel, unpredictable issue?
Hint: Consider the distinction between verifying expected behavior on a dashboard and investigating entirely unknown failure modes.
  • Monitoring
  • High-cardinality Logging
  • Observability
  • Alertmanager Grouping
View Correct Answer & Rationale
Correct Answer: Observability

Slicing raw signals at query time allows engineers to ask new questions about unexpected behaviors without deploying new code. Monitoring watches known indicators; Logging alone lacks combined querying across signals; Alertmanager is for notifications.
2. A customer claims their shopping cart emptied itself right after they clicked 'Submit Order.' Which observability pillar is best suited to determine exactly what actions this specific user took?
Hint: If you need a step-by-step audit trail of user actions, think about the system's 'diary'.
  • Logs
  • Metrics
  • Traces
  • PromQL
View Correct Answer & Rationale
Correct Answer: Logs

Discrete, time-stamped events provide the necessary audit trail and forensics to reconstruct individual user sessions. Metrics aggregate numbers; Traces track latency call graphs; PromQL queries metrics.
3. A PagerDuty alert fires because the p99 latency has spiked. To investigate, an engineer needs to find the exact microservice causing the bottleneck. Which mechanism bridges the gap between the latency spike and the specific call graph?
Hint: Look for the element that acts as a unique identifier passed between services to map the entire request journey.
  • Syslog facilities
  • Correlation IDs (trace_id)
  • Prometheus Exporters
  • Alertmanager inhibition rules
View Correct Answer & Rationale
Correct Answer: Correlation IDs (trace_id)

A unique identifier propagated through the system allows an engineer to pivot from a high-level metric alert to the specific path of a failing request.
4. An application running as a systemd unit on a modern Linux distribution fails to start. What is the most direct command to view the structured, binary records associated exclusively with this application?
Hint: Focus on the command-line tool designed specifically to interact with systemd's local log collection service.
  • tail -f /var/log/syslog
  • journalctl -k
  • systemctl status [unit name]
  • journalctl -u [unit name]
View Correct Answer & Rationale
Correct Answer: journalctl -u [unit name]

This command specifically queries the systemd journal to retrieve logs isolated to a single, specified service unit.
5. In the Prometheus architecture, the central server periodically scrapes an HTTP endpoint on every target. What is the primary architectural advantage of this approach in a dynamic environment?
Hint: Consider the simplest way a central system can determine if a remote microservice has completely crashed.
  • It allows clients to implement back-pressure to protect the central server from traffic spikes.
  • It inherently acts as a health check, meaning if a scrape fails, the target is known to be down.
  • It requires targets to use a heavy SDK to push metrics over the network.
  • It bypasses the need for service discovery mechanisms like Consul or Kubernetes.
View Correct Answer & Rationale
Correct Answer: It inherently acts as a health check...

Relying on the server to retrieve data removes the need for separate liveness checks, as unresponsiveness directly indicates a failure.
6. A developer wants to track the total number of HTTP requests handled by an application since it was last deployed. Which metric type should be used, and how should it typically be queried?
Hint: Compare a metric that acts like a car's speedometer with one that acts like its odometer, accumulating distance over time.
  • A Gauge, which fluctuates up and down and is queried as an absolute value.
  • A Counter, which only goes up and is typically queried using a rate() function.
  • A Histogram, which calculates percentiles on the server side based on bucketed observations.
  • A Summary, which pre-computes quantiles on the client side before scraping.
View Correct Answer & Rationale
Correct Answer: A Counter...

Tracking cumulative events that reset to zero on restart requires a monotonically increasing metric, mathematically converted to a per-second value using rate() for dashboards.
7. To create a highly detailed dashboard, an engineer adds the user's universally unique identifier (UUID) as a label to the `http_requests_total` metric. What is the most likely consequence of this action?
Hint: Consider how a time-series database organizes dimensions and what happens when the number of distinct tracking categories approaches infinity.
  • It will automatically convert the metric from a Counter into a Histogram.
  • It will seamlessly allow Grafana to filter traffic by individual users.
  • It will cause a cardinality explosion that crashes the Time-Series Database (TSDB).
  • It will trigger an Alertmanager inhibition rule due to excessive data generation.
View Correct Answer & Rationale
Correct Answer: It will cause a cardinality explosion...

Storing a unique time-series for every possible permutation of unbounded values rapidly exhausts the memory and storage capacity of the monitoring backend.
8. When designing the top row of a new Grafana dashboard for a critical service, what information should be prioritized for immediate visibility?
Hint: Apply the 'hierarchy of usefulness'—imagine opening a dashboard at 3 AM and needing an immediate assessment of the situation.
  • The RED metrics (Rate, Errors, Duration) broken down by individual endpoints.
  • Highly specific drill-downs, such as per-pod CPU utilization.
  • A continuously scrolling feed of the raw application logs from the journald service.
  • A high-level, traffic-light colored indicator answering if the overall system is okay.
View Correct Answer & Rationale
Correct Answer: A high-level, traffic-light colored indicator...

Information positioned at the highest visual hierarchy should provide an immediate, 10-second assessment of system health before requiring further investigation into deeper metrics.
9. An alert is configured to page an on-call engineer at 3 AM if the background job queue depth exceeds 500. However, the jobs are processing normally without customer impact, and the queue naturally drains by 5 AM. Why is this considered a poorly designed alert?
Hint: Evaluate the scenario against the core rules of alerting, specifically regarding the direct impact on the end user.
  • It should trigger based on CPU saturation rather than the depth of the queue.
  • It uses a Gauge metric when it should be using a Counter metric to track the queue.
  • It is not symptom-based and does not represent actionable, user-visible pain.
  • It lacks a correlation ID to tie the queue depth to a specific trace.
View Correct Answer & Rationale
Correct Answer: It is not symptom-based...

Notifications that wake a human must be tied to actual degradation of service, otherwise they train engineers to ignore alerts and contribute to alert fatigue.
10. According to standard severity conventions, an alert fires indicating that an SSL certificate will expire in 14 days. How should Alertmanager ideally route this 'warning' severity alert?
Hint: Align the severity 'warning' with the standard routing convention for issues that are pending but not yet causing immediate pain.
  • To PagerDuty, because certificate expiration will eventually cause a total system outage.
  • To a team Slack channel, as the issue requires attention before it becomes a critical failure.
  • To an email digest, because the issue is purely informational and requires no human action.
  • To an inhibition rule, ensuring the alert is suppressed until the certificate actually expires.
View Correct Answer & Rationale
Correct Answer: To a team Slack channel...

Issues that indicate impending, but not immediate, pain are best suited for asynchronous team communication platforms rather than high-urgency pagers designed for active outages.
